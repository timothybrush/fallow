# Detection capabilities

Non-obvious implementation details for each detection feature. These are NOT discoverable from reading the code alone.

## Parser-level
- **HTML entry files**: `<script src="...">` (module and classic) and `<link rel="stylesheet" href="...">` / `<link rel="modulepreload" href="...">` create graph edges from HTML to referenced assets. Remote URLs skipped. HTML comments stripped before matching. HTML files exempt from unused-file detection.
- **JSX resource attributes are ignored**: generic lowercase intrinsic `<script src="...">` and `<link rel="stylesheet|modulepreload" href="...">` inside JSX/TSX files do NOT create `SideEffect` imports by default. These literals are often runtime HTML metadata or serializer test values, not bundler-resolved module specifiers, and treating them as imports caused false `unresolved-imports` clusters in Hono's JSX tests. Framework-specific JSX asset semantics should be added as constrained opt-in logic rather than broad JSX attribute extraction. HTML files and bare `html` tagged templates still have dedicated asset extraction. See issues #105 and #640.
- **`html` tagged template asset references**: `` html`<script src="..."></script>` `` and `` html`<link rel="stylesheet|modulepreload" href="..."/>` `` inside any JS/TS file create `SideEffect` imports via the regex-based HTML asset scanner (`crate::html::collect_asset_refs`). Only the bare identifier tag `html` is matched — `css`, `sql`, `gql`, `styled.div`, and member/call expressions are deliberately ignored to avoid misreading unrelated tagged templates as HTML. Each template quasi is scanned independently, so an asset reference split across an interpolation boundary (`` html`<script src="${base}/app.js">` ``) is skipped rather than producing a garbled unresolvable specifier. Needed for SSR helpers like `hono/html`, `lit-html`, and `htm` where layout components emit HTML via tagged templates instead of JSX. See issue #105 (till's follow-up comment).
- **GraphQL document imports**: `.graphql` and `.gql` files are discovered as source files. Lines matching `#import "./fragment.graphql"` or `# import '../shared/fragment.gql'` create `SideEffect` imports so fragment/schema documents reachable only through GraphQL import comments stay connected in the module graph. Only relative string specifiers (`./`, `../`) are extracted; package-style imports are ignored. Extensionless relative imports probe `.graphql` and `.gql` through the resolver extension list. See issue #250.
- **MDX import/export extraction**: `.mdx` files extract only top-level JavaScript/TypeScript `import` and `export` statements, including multi-line brace imports. Fenced Markdown code blocks using backtick or tilde fences are skipped entirely, so documentation examples with virtual `// file:` boundaries do not create graph edges or `unresolved-import` findings. Extracted statement bytes carry a source map back to the original MDX file, so diagnostics on imports after prose report the original line and column rather than the compacted synthetic buffer line. Real top-level MDX imports remain analyzed. See issues #639 and #445.
- **Fumadocs MDX content roots**: the Fumadocs plugin activates on `fumadocs-mdx`, `fumadocs-core`, `fumadocs-ui`, or `source.config.*`. It treats `source.config.*` and generated `.source/**/*.{ts,tsx,js,jsx,mts,mjs,cts,cjs}` files as framework-used, traverses `.source` despite the hidden-directory default skip, suppresses generated `fumadocs-mdx:*` virtual imports, credits packages imported by `source.config.*`, and extracts literal `dir` values from `defineCollections`, `defineDocs`, and direct `defineConfig({ collections })` object entries as Markdown/MDX/JSON/YAML content roots. This prevents docs content under configured roots from surfacing as unused files while avoiding a broad `content/**` fallback. See issue #633.
- **Contentlayer content roots**: the Contentlayer plugin activates on `contentlayer`, `contentlayer2`, `next-contentlayer`, `next-contentlayer2`, or `contentlayer.config.{ts,js,mts,mjs}`. It treats the config and generated `.contentlayer/**/*.{ts,tsx,js,jsx,mts,mjs,cts,cjs}` files as framework-used, traverses `.contentlayer` only for active packages, credits packages imported by the config, and extracts static `contentDirPath` plus same-file `defineDocumentType` `filePathPattern` literals as Markdown/MDX/JSON/YAML content roots. Dynamic config and cross-file document type evaluation are intentionally skipped to avoid over-crediting. See issue #610.
- **Firebase Messaging service worker**: the Firebase plugin activates on the exact `firebase` dependency and marks `public/firebase-messaging-sw.js` plus nested `**/public/firebase-messaging-sw.js` files as always used. It does not credit the `firebase` package by itself, so unused dependency reporting still depends on source imports or other package usage. See issue #873.
- **Velite content roots**: the Velite plugin activates on the `velite` dependency or a root/workspace `velite.config.{ts,mts,cts,js,mjs,cjs}` file. It keeps the config and generated `.velite/**` output always-used, traverses the hidden `.velite` directory for active packages so app imports of generated collections resolve, credits packages imported by the config and `velite` itself as a tooling dependency, and parses `defineConfig` / `defineCollection` to model content roots. Each collection `pattern` glob (string or array) is joined to the top-level `root` (default `content`, resolved config-relative) and emitted as a support entry point, so Velite-managed Markdown / MDX content is not reported as unused; negation globs (`!private/**`) are skipped and a config with no recoverable pattern falls back to `<root>/**/*.{md,mdx,yml,yaml,json}`. A non-default `output.data` directory is credited config-relative. Patterns are scoped to the config file's package, so a monorepo app's content does not leak project-wide. See issue #609.
- **Mintlify docs content roots**: the Mintlify plugin activates on the `mint` or `mintlify` dependency, or a root/workspace `docs.json` or `mint.json` file. It keeps `docs.json` / `mint.json` always-used, credits `mint` and `mintlify` as tooling dependencies (so the CLI does not surface as unused even when invoked only through a `mint dev` script), and marks `{md,mdx}` under the directory that holds the config file as runtime-used support entry points. The content pattern is scoped to the config file's directory (workspace runs prefix it back to the project root), so docs MDX no longer needs manual `dynamicallyLoaded` entries while non-Mintlify MDX outside the docs root stays governed by other plugins. The whole docs-root tree is credited rather than parsing the `navigation` graph, which avoids false positives across Mintlify's many navigation shapes. See issue #626.
- **RedwoodSDK worker entrypoints**: the RedwoodSDK plugin activates only from `rwsdk` and marks `src/worker.{ts,tsx,js,jsx,mts,mjs}` as runtime entrypoints for that workspace. It does not claim `vite.config.*` because the existing Vite plugin owns Vite config reachability and import-credit parsing. The worker convention is workspace-scoped, so plain Vite sibling workspaces with their own `src/worker.tsx` do not inherit it. Legacy `@redwoodjs/*` packages do not activate this plugin. See issue #632.
- **OpenNext Cloudflare config**: the OpenNext Cloudflare plugin activates from `@opennextjs/cloudflare` in package dependencies or devDependencies, or from package scripts invoking `opennextjs-cloudflare`. It keeps root and nested `open-next.config.{ts,js,mjs,cjs}` files reachable and credits packages imported from those configs, matching `opennextjs-cloudflare build` consumption without broadening plain Next.js or Wrangler projects. See issue #613.
- **React Compiler dependency credit**: `babel-plugin-react-compiler` is credited only when React Compiler is explicitly enabled by framework config. Next.js `next.config.*` credits it for `reactCompiler: true` and object-valued `reactCompiler` options. Vite `vite.config.*` and electron-vite `electron.vite.config.*` credit it when it is wired through `@vitejs/plugin-react` or the standalone `@rolldown/plugin-babel`: explicit Babel plugin strings and tuples in `react({ babel: { plugins } })` / `babel({ ... })`, and the documented `reactCompilerPreset()` preset helper inside a `presets` array (e.g. `babel({ presets: [reactCompilerPreset()] })`, the canonical `@vitejs/plugin-react` 6.x shape). The preset call is credited only when `reactCompilerPreset` is a named import from `@vitejs/plugin-react` (provenance-checked) and the call sits directly inside a `presets` / `babel.presets` array of a `react()` / `babel()` plugin call; for electron-vite the same arrays are scanned inside the `main` / `preload` / `renderer` sections (the Vite plugin never sees `electron.vite.config.*`). The detection lives in the shared `crate::plugins::react_compiler` module used by both the Vite and Electron plugins. Disabled config, missing config, local functions with matching names, the namespace-import form (`import * as vr; vr.reactCompilerPreset()`), the variable-indirection form (`const p = reactCompilerPreset()`), a local re-export barrel that does not import from `@vitejs/plugin-react`, and arbitrary nested strings do not credit the dependency. See issues #623 and #751.
- **Storybook staticDirs and manager runtime**: the Storybook plugin parses statically recoverable `staticDirs` entries from `.storybook/main.{ts,js,mjs,cjs}`. String entries mount at `/`; object entries use `from` as a config-relative source directory and normalize `to` values such as `icons`, `/icons/`, and `/`. Resolver handling is intentionally limited to `.storybook/preview-head.html` and `.storybook/preview-body.html`; normalized HTML refs such as `tokens.css` / `./tokens.css` are treated as URL-root paths, query/hash suffixes are stripped for filesystem lookup, and the longest matching mount wins before the generic root-relative URL fallback. Mapped project files become graph edges and external existing files, such as files under `node_modules`, resolve as `ExternalFile` without being synthesized back into `unlisted-dependencies` findings. `.storybook/manager.{ts,tsx,js,jsx}` also has file-scoped runtime dependency credit for Storybook manager-provided imports (`react`, `react-dom/client`, `@emotion/*`, `storybook/manager-api`, `storybook/internal/*`, `storybook/theming`, `storybook/core-events`, `@storybook/components`, `@storybook/theming`). This is matched against original import specifiers so `storybook/internal/*` does not suppress every `storybook/*` import, and the allowance does not apply to ordinary source files. See issue #546.
- **SvelteKit remote functions**: SvelteKit 2.27+ remote functions are exported from `*.remote.{ts,js}` files (`export const getPosts = query(...)`, plus `command` / `form` / `prerender`) and invoked through framework-generated client/server bindings that fallow's import graph cannot see. The SvelteKit plugin marks `src/**/*.remote.{ts,js}` as an entry pattern (so a generated-binding-only remote file is not reported as `unused-file`) and adds a `used_exports` rule mapping the same pattern to `["*"]` (so the user-named remote-function exports are credited and never report as `unused-export`, including under `--include-entry-exports` where the entry-point skip is disabled). Remote-function export names are user-defined, so a fixed convention list cannot enumerate them; `"*"` is the conservative credit, mirroring the Storybook story-file precedent. Import edges for dependencies referenced inside remote files are preserved because the file is parsed as an entry point. Scoping is by the `.remote.` double-extension only, so ordinary `src/**/*.ts` files (and a genuinely-unused non-remote module) stay governed by normal detection; `src/lib/server/**` is not excluded because SvelteKit forbids remote files there and crediting a stray one is harmless. See issue #611.
- **SvelteKit layout-reset routes**: SvelteKit's layout-reset convention appends an `@` suffix to a page or layout component to break it out of its parent layout chain (`+page@.svelte` resets to the root layout, `+page@named.svelte` resets to a named layout). The SvelteKit plugin marks `src/routes/**/+page@*.svelte` and `src/routes/**/+layout@*.svelte` as entry patterns and adds `used_exports` rules crediting their `default` export, so layout-reset routes stop reporting as `unused-file` and their component default stays credited. The `*` covers both the empty (`@.svelte`) and named (`@named.svelte`) forms; `literal_separator(true)` keeps the glob within a single path segment. Only `.svelte` component files take the suffix: SvelteKit itself throws on `@` in `.ts`/`.js` route files (`+page@.ts` is invalid), and `+error.svelte` / `+server.{ts,js}` have no `@` variant, so no module-file or error/server patterns are added. See issue #791.
- **electron-vite `rollupOptions.input` entries**: the Electron plugin activates on `electron` / `electron-builder` / `@electron-forge/cli` / `electron-vite`. When `electron.vite.config.{ts,js,mjs}` is present it now parses each of the `main` / `preload` / `renderer` sections' `build.rollupOptions.input` (string, array, and object-of-values forms) and seeds each value as an entry point, normalized relative to the config file via `normalize_config_path` (so a monorepo subpackage config maps `src/renderer/index.html` to `apps/desktop/src/renderer/index.html`). Object values commonly use path helpers (`resolve(__dirname, 'src/renderer/index.html')`, `join(...)`, `fileURLToPath(...)`, `new URL(..., import.meta.url)`); the shared `config_parser::extract_config_string_or_array` evaluates those through `expression_to_path_string` in string / array / object positions (broadened in issue #604), so the Electron plugin reuses it directly rather than carrying its own extractor. Renderer HTML entries become entry points, so the `<script src>` source trees behind multi-window renderers stop reporting as `unused-files`. The static main/preload fallback globs also widen to `{ts,tsx,js,jsx,mts,mjs}`. Crediting is scoped to declared inputs (and per-section function-wrapped configs such as `renderer: defineConfig({...})` are out of scope, matching the Vite plugin's plain-object nesting requirement), so a renderer source file linked from no declared entry is still reported. See issue #600.
- **Bun `bunfig.toml` preload entries**: the `bun` plugin now activates on `@types/bun` in addition to `bun-types` (activation reads declared `package.json` deps, and `@types/bun` is the canonical Bun types package even though it transitively depends on `bun-types`). When `bunfig.toml` is present the plugin parses both the top-level `preload = [...]` array and the `[test] preload = [...]` array and seeds each referenced file as an entry pattern, normalized relative to the config file via `config_parser::normalize_config_path` (so a `./src/test-preload.ts` reference maps to `src/test-preload.ts`). Bun loads these files before every run / test without an inbound import edge, so without the seed they report as `unused-file`. Invalid TOML and non-array / non-string entries are skipped silently; entries are deduped. Both `bun-types` and `@types/bun` are credited as tooling dependencies. See issue #847.
- **Browser extension manifest entrypoints**: the `browser-extension` plugin activates without package dependencies by inspecting deduped `manifest.json` candidates at the project root and discovered-file ancestor directories. Activation requires numeric `manifest_version` 2 or 3 plus at least one extension runtime key (`background`, `content_scripts`, action/page/browser action, options, side panel, devtools, or `web_accessible_resources`) so ordinary PWA manifests stay inert. Config resolution seeds MV3 `background.service_worker`, MV2 `background.scripts`, `content_scripts[].js/css`, popup/options/devtools/side-panel HTML, MV2 string-array `web_accessible_resources`, and MV3 object `resources` arrays as runtime entry patterns, normalized relative to the manifest file. Remote/scheme URLs, protocol-relative URLs, empty strings, and parent-directory escapes are skipped; leading `/foo.js` is treated as extension-root-relative. HTML entries rely on the existing HTML asset extractor to keep their `<script>` and stylesheet graph reachable. See issue #616.
- **WXT entrypoints and config modules**: the `wxt` plugin activates from `wxt`, any non-empty `@wxt-dev/*` dependency, or a root `wxt.config.{ts,js,mts,mjs,cjs,cts}` file. It keeps WXT config files always used, credits their default export under `--include-entry-exports`, marks documented single-file `entrypoints/*` and directory `entrypoints/**/index.*` entrypoints (plus `src/entrypoints` variants) as runtime roots, and parses static `modules: [...]` strings from config as referenced dependencies. Package subpaths are normalized (`@wxt-dev/i18n/module` credits `@wxt-dev/i18n`); local, absolute, protocol, empty, and backslash-containing specifiers are ignored. The entry globs intentionally do not credit every descendant under an entrypoint directory, so unimported helper siblings stay reportable. See issue #612.
- **@sanity/pkg-utils build configs**: the `pkg-utils` plugin activates ONLY on an exact `@sanity/pkg-utils` dependency (not the broad `@sanity/` prefix the CMS `sanity` plugin uses, so a plain `@sanity/client` consumer does not get its build config auto-credited). `@sanity/pkg-utils` discovers `package.config.{ts,js,mts,mjs,cts,cjs}` (its build config) and the optional `package.bundle.{ts,js,mts,mjs,cts,cjs}` (Vite bundle config) on a filename convention rather than importing them from source, so without the plugin they surface as `unused-file`. The plugin marks both filenames as always-used at the project root AND nested (each pattern listed in bare and `**/`-prefixed form, because `always_used` matching uses `literal_separator(true)` with no automatic prefix; mirrors the `varlock` / `wuchale` precedent), covering monorepo `packages/<pkg>/package.config.ts` layouts, and credits `@sanity/pkg-utils` as a tooling dependency. It does not seed library source entries (`src/_exports/**`): the `exports.source` condition resolution plus the workspace-package fallback already keep them reachable, so parsing `package.bundle.ts`'s `build.lib.entry` would be redundant. The credit is scoped to the two build-config filenames, so an unreferenced ordinary source file still reports. Lets pkg-utils projects drop the manual `entry` array for these files (adopter: sanity-io/sdk).
- **Obsidian plugin entrypoints and lifecycle members**: the `obsidian` plugin activates from an `obsidian` dependency or a conservative Obsidian-shaped `manifest.json` (`id`, `name`, `version`, and `minAppVersion` as strings, with no browser-extension `manifest_version`). It marks `src/main.{ts,js}`, `main.{ts,js}`, and root-level `cdp.js` as runtime entrypoints, keeps `manifest.json` and `styles.css` reachable, and credits documented lifecycle overrides only on direct Obsidian API bases: `Plugin.onload` / `Plugin.onunload`, `Modal.onOpen` / `Modal.onClose`, and `ItemView` / `View` methods `getViewType`, `getDisplayText`, `getIcon`, `onOpen`, `onClose`, and `onPaneMenu`. Aliased imports and project-local transitive base classes are intentionally not resolved by the scoped member rule, so similarly named methods on ordinary classes still report. See issue #617.
- **Oxlint `jsPlugins` config dependencies**: the Oxlint plugin activates on `oxlint`, treats `.oxlintrc.json`, `oxlint.json`, and `oxlint.config.ts` as config files, and parses top-level `jsPlugins` entries as referenced dependencies. String entries and tuple first elements credit package specifiers via package-name extraction without ESLint shorthand expansion; alias-object entries credit the `specifier` value, not the display `name`; relative or absolute plugin paths are recorded as support setup files instead of package dependencies. Built-in `plugins` names remain ignored because they are Oxlint-native plugin selectors, not npm package references. The plugin also credits known oxlint CLI tooling packages by exact name via `tooling_dependencies`: `oxlint-tsgolint` (the type-aware companion the oxlint binary loads at runtime via `--type-aware` / `options.typeAware`, never imported and never in `jsPlugins`). This credit is honored for both prod and dev dependency categories, so an `oxlint-tsgolint` declared in prod `dependencies` (where the general `oxlint` tooling prefix, which only covers devDependencies, does not apply) is no longer reported as unused. The list is exact-name, not an `oxlint-*` prefix, so an unrelated `oxlint-`-prefixed prod dependency still reports. See issues #607 and #753.
- **ESLint meta-preset plugin crediting**: meta-presets such as `@antfu/eslint-config` are invoked as a factory (`antfu({...})`) and load their plugins dynamically or via optional `peerDependencies`, so the flat-config `plugins` object-key credit names nothing and the generic required-peer-dependency closure (`required_peer_dependency_names`, which skips optional peers) leaves the framework plugins reported as unused. When the ESLint plugin sees an import whose package name matches a preset shape (`eslint-config-*` or any `@scope/eslint-config[-*]`, detected by `is_eslint_preset`), it reads that preset's own `package.json` from node_modules and credits every entry in `dependencies` / `peerDependencies` / `optionalDependencies` that is eslint-ecosystem-shaped (`is_eslint_ecosystem_dependency`: `eslint-plugin-*`, `@scope/eslint-plugin[-*]`, `eslint-config-*`, `@scope/eslint-config`, `eslint-import-resolver-*`, `*-eslint-parser`, `@scope/parser`). General-purpose runtime deps the preset also declares (`globals`, `ansis`, `cac`) are excluded so a genuinely-unused general dep the user declared independently is not masked. Crediting is one level deep and gated to preset imports, so a plain plugin import (`eslint-plugin-import`) does not pull in its own dependency fan. See issue #754.
- **Package scaffold template assets**: `package.json#files` entries that clearly name runtime-copied template or scaffold roots are treated as support entry patterns outside framework plugin activation. Supported shapes include `template-*`, `templates`, `templates/**`, `scaffold/**`, and `scaffolds/*`, with workspace package patterns prefixed from the monorepo root. Generic publish entries such as `dist`, `index.js`, and `README.md` are ignored, and packages with negated `files` entries skip this derivation because support entry patterns do not model exclusions. See issue #635.
- **OpenCode project plugins**: the OpenCode plugin activates on `opencode.json`, `.opencode/`, or any `@opencode-ai/*` dependency. It traverses `.opencode` despite the hidden-directory default skip, treats `.opencode/plugins/**/*.{js,ts,mjs,cjs,mts,cts}` as support entry files, keeps `opencode.json` and `.opencode/package.json` reachable, and credits npm plugin packages declared in `opencode.json#plugin`. String entries and tuple first elements such as `["opencode-wakatime", { ... }]` are normalized through package-name extraction; local, absolute, protocol-like, empty, and backslash-containing specifiers are ignored for package credit. `@opencode-ai/plugin` is also treated as a tooling dependency so local plugin type imports do not surface as unused. See issue #629.
- **Varlock schema plugins**: the Varlock plugin activates on `varlock`, any `@varlock/*` dependency, a root `.env.schema`, or discovered nested `.env.schema` files. It keeps `.env.schema` files reachable, credits package-like `@plugin(...)` schema decorator arguments as referenced dependencies, strips npm version selectors before matching, and treats `varlock` plus `@varlock/vite-integration` as tooling dependencies. Local paths, URL/protocol references, malformed values, and backslash-containing specifiers are ignored for dependency credit. See issue #622.
- **k6 load-test scripts**: the k6 plugin activates from a `k6` dependency or any discovered `*.k6.{js,ts,mjs,cjs,mts,cts}` file. It marks `**/*.k6.{js,ts,mjs,cjs,mts,cts}` and `load/*.k6.{js,ts,mjs,cjs,mts,cts}` as test entry points so load-test scripts do not report as unused, and treats `k6` as a tooling dependency when package scripts invoke the CLI. Runtime imports are handled separately by the platform-builtin predicate: exact `k6` and `k6/*` are provided by the k6 runtime, not npm resolution. Similar package names such as `k6-tools`, `k6ish`, `k6x/http`, and `@types/k6` stay ordinary dependencies. See issue #625.
- **Danger dangerfiles**: the Danger plugin activates from a `danger` dependency OR from any discovered `dangerfile.{js,ts,mjs,cjs}` (root or workspace). Danger is commonly run straight from CI (`npx danger ci`, an Earthfile, a GitHub Action) without ever listing `danger` in package.json, so the dependency enabler alone left a root `dangerfile.js` reported as `unused-file`. It keeps `dangerfile.{js,ts,mjs,cjs}` (and the `**/`-prefixed workspace form) always-used and credits `danger` as a tooling dependency when declared. Activation uses an exact filename match, so `predangerfile.js`, `dangerfile.json`, and `dangerfiles.js` stay inert. The `.js` extension was always in the allowlist (globset expands the `{...}` alternation); the gap was activation, not the extension. See issue #758.
- **JSDoc `import()` type references**: JSDoc comments matching `import('./path').Member` inside `@param`, `@returns`, `@type`, `@typedef`, `@callback`, or other tag contexts are extracted as type-only imports. Supports single/double quotes, bare specifiers, parent-relative paths, union annotations with multiple `import()` expressions per comment, and nested member access (first segment wins). Side-effect-only `import('./types')` without `.Member` is recorded as `SideEffect` to keep the file reachable. Prevents false `unused-export` reports for types referenced only from JSDoc annotations in untyped JS files. Only `/** */` block comments are scanned (single-star `/* */` is not JSDoc). See issue #105 (till's comment).
- **`typeof import('./path').X` in TypeScript type positions**: `TSImportType` AST nodes are extracted as type-only `ImportInfo` entries. A flat qualifier (`typeof import('./x').foo`) credits a `Named("foo")` import; a nested qualifier (`typeof import('./x').A.B`) walks to the leftmost identifier and credits `Named("A")`; no qualifier and the indexed-access form (`typeof import('./x')['default']`) record a `SideEffect` import so the target file stays reachable. Fires inside any type position, including the bodies of `declare global { ... }` and `declare module 'pkg' { ... }` ambient declarations produced by unplugin-auto-import (`auto-imports.d.ts`) and unplugin-vue-components (`components.d.ts`). Pairs with the graph-level promotion of every `.d.ts` / `.d.mts` / `.d.cts` file to an overall reachability root (`ModuleGraph::build_with_reachability_roots` augments `entry_point_ids` with declaration files), so the new edges propagate through BFS even when the d.ts file is not in the user's `entry` config. Side effect: `is_entry_point()` returns `true` for declaration files, which auto-skips them in `unused-exports` / `unused-types` detection (preventing false positives on module-augmentation interfaces TypeScript consumes via global type resolution); pass `--include-entry-exports` to re-enable. Runtime / test reachability is unchanged because declaration files have no runtime side effects. See issues #396 and #397.
- **`new URL('./', import.meta.url)` directory specifiers skipped**: the worker / asset detector for `new URL(literal, import.meta.url)` admits relative specifiers (`./`, `../` prefixes) but rejects trailing-slash forms (`./`, `../`, `./foo/`) and marks extensionless specifiers (e.g. `./services`, `./bin`) as speculative. Trailing-slash forms were excluded in #399 because they are definitionally directory URLs. Extensionless specifiers added in #840: they lack a file extension and are commonly used to obtain a directory URL (`fileURLToPath(new URL('./services', import.meta.url))`); when the resolver cannot find a module at those paths they are silently dropped rather than reported as `unresolved-import`. File-pointing specifiers with an extension (`./worker.js`, `./assets/foo.svg`) keep `is_speculative = false` so genuinely missing files are still reported. See issues #399 and #840.
- **Vue/Svelte SFC**: handles `>` in quoted attributes like `generic="T extends Foo<Bar>"`, Vue `<script src="...">` external script support, Svelte script `src` metadata parsing, and HTML comment filtering. Script bodies are parsed as extracted fragments whose spans are remapped back to original `.vue` / `.svelte` byte offsets before module info is finalized; Vue `<script src>` and style `src` references carry the `src` value span when available. Svelte markup-level `<script src>` references do not create graph edges because Svelte treats them as runtime HTML. Template-visible import tracking: imports used only in markup (`{formatDate(x)}`, `utils.formatDate()`) are credited as used, preventing false unused-import/export reports. Vue credits only `<script setup>` bindings; Svelte excludes `context="module"` scripts from template visibility. Namespace member access in templates (`utils.formatDate`) tracked as member usage. Svelte 5 `$derived(new Class())` / `$derived.by(() => new Class())` bindings propagate template member access to the derived class instance, while `$effect(...)` and `$inspect(...)` are ordinary script calls whose referenced imports are credited by the JS visitor. See issues #445 and #835.
- **CSS Modules class extraction skips `@layer` and `@import` preludes**: `extract_css_module_exports` runs `CSS_AT_RULE_PRELUDE_RE = @(?:layer|import)\b[^;{]*` over the source after string and `url()` stripping and before the class regex, so dot-separated cascade-layer names (`@layer foo.bar { ... }`, `@layer foo.bar, foo.baz;`) and parenthesised layer references on `@import` (`@import url("x.css") layer(theme.button)`) never reach the class scan. - **Glimmer component files**: `.gts` and `.gjs` files are discovered as source files. `.gts` parses with TypeScript source semantics and `.gjs` parses with JavaScript module semantics. `<template>...</template>` blocks are stripped before Oxc parsing while preserving byte offsets and line numbers, so imports before/around templates are still extracted and resolved through the same tsconfig path-alias path as `.ts` importers. Two replacement strategies are used so the post-strip source stays syntactically valid in both canonical GTS contexts: class-body templates (`class X { <template>...</template> }`) blank to spaces, leaving an empty class body; module-level template expressions (`const X = <template>...</template>`) and other expression-position templates (function argument, ternary branch, decorator-call argument) are replaced with a byte-length-preserving parenthesized template literal `` (`...`) ``. - **Playwright POM fixtures**: typed `base.extend<MyFixtures>({ ... })` definitions are correlated with their callback-side member uses so POM class methods consumed only through fixtures are credited. ## Resolution-level
- **Package.json `exports` subpath**: output dirs (dist/build/out/esm/cjs) mapped back to src/ with source extension fallback, including nested subdirs
- **Pnpm virtual store**: `.pnpm` paths mapped back to workspace source files. Handles injected deps, scoped/unscoped packages, peer dependency suffixes.
- **Package.json `imports` (`#subpath`)**: simple mappings, wildcard patterns, conditional exports. Per-package scoping.
- **Export / import condition names**: `oxc_resolver` is configured with the baseline condition set `["development", "import", "require", "default", "types", "node"]`. `"development"` ships unconditionally so packages that declare a `development` branch in `exports` / `imports` (common pattern: source files for dev, compiled output for prod; used by Vite, Vitest, esbuild, Rollup) resolve deterministically to source. When the React Native or Expo plugin is active, `"react-native"` and `"browser"` are prepended. Users extend the list via the `resolve.conditions` config key; those entries prepend ahead of everything else. See issue #135 and <https://nodejs.org/api/packages.html#community-conditions-definitions>.
- **React Native platform extensions**: `.web.ts`, `.ios.ts`, `.android.ts`, `.native.ts` resolved alongside standard extensions
- **Web-root-relative paths**: `/src/main.tsx` or `/static/style.css` resolved as a web convention (Vite/Parcel/Hono). Converts to `./src/main.tsx` and resolves from the source file's parent directory first, then falls back to project root. This handles monorepo workspace members where `site/index.html` referencing `/src/main.tsx` resolves to `site/src/main.tsx`, not `<root>/src/main.tsx`. Applied to HTML, JSX, TSX, JS/TS, and Glimmer source files (extensions `.html`, `.jsx`, `.tsx`, `.js`, `.ts`, `.mjs`, `.cjs`, `.mts`, `.cts`, `.gts`, `.gjs`) because HTML-like asset extraction can originate from HTML files or from `html` tagged templates in JS/TS-family files. Generic JSX resource attributes are intentionally ignored by default. See issues #105 and #640.
- **CSS scoped package imports**: CSS `@import` with scoped npm packages (`@fontsource/monaspace-neon/400.css`, `@company/design-system/styles.css`) are kept as bare specifiers for node_modules resolution. Unscoped bare imports with CSS/SCSS extensions are normalized to relative paths (`app.css` becomes `./app.css`).
- **External Sass package entrypoints**: When a project stylesheet imports a package stylesheet through a package `sass` or `style` export, the external style scanner credits the owner package and recursively follows package-local relative `@use` / `@forward` / `@import` imports from that external file. The fallback is only active while scanning an existing stylesheet under `node_modules`, and uses Sass-style extension, underscore partial, and `_index` / `index` lookup, so imports like `@use "@angular/material" as mat;` do not cascade into `unresolved-import` or `unused-dependency` findings when Angular Material forwards package-local SCSS files. Missing project-local SCSS imports still report normally. See issue #871.
- **Tailwind CSS `@plugin` directives**: CSS/SCSS `@plugin "pkg"` targets are extracted as default imports so package dependencies are credited and relative plugin modules have their default export marked used. Unlike SCSS `@use`, extensionless package names such as `daisyui` stay bare even in `.scss`/`.sass` files; otherwise they would be misread as local partials.
- **Iconify static icon strings credit `@iconify-json/<prefix>` packages**: icon components consume an icon set through a build-time string name (`<Icon name="jam:github" />`, `<List icon="ic:round-home" />`) rather than a JavaScript import, so the `@iconify-json/<prefix>` package supplying that collection was reported as an unused dependency. The extraction layer scans markup file kinds (`.astro`, `.jsx`, `.tsx`, `.svelte`, `.vue`, `.html`/`.htm`, `.mdx`) for static `icon=` / `name=` props whose value is `<prefix>:<name>` (HTML comments stripped first; capture is the collection prefix only) and records the deduped prefixes on `ModuleInfo.iconify_prefixes`. The analysis bridge (`crates/core/src/analyze/iconify.rs`) maps each prefix to `@iconify-json/<prefix>` and adds it to `referenced_dependencies`, GATED on the project declaring an Iconify-ecosystem dependency (`astro-icon`, `unplugin-icons`, `@iconify/*`, `@iconify-json/*`, `@iconify-icons/*`) across the root or any workspace `package.json`. Because the credit only exempts an already-declared package from the unused report and never creates a finding, a stray non-icon `name="foo:bar"` is harmless. Plain `.js`/`.ts` files, dynamic bindings (`:name=`, `name={expr}`), JSX brace-string form, and icon names referenced only from `.ts` data objects (`icon: "mdi:home"`) are out of scope. Bumps the extract `CACHE_VERSION` because the new `iconify_prefixes` field changes serialized module state. See issue #608.
- **Tsconfig path aliases**: per-file discovery resolves `@/utils` by finding nearest tsconfig.json per file
- **Broken tsconfig chain fallback**: when `resolve_file` fails with a tsconfig-loading error (broken `extends` chain, circular extend, self-referencing project reference, malformed JSON, IO error), fall back to `resolve(dir, specifier)` which bypasses `TsconfigDiscovery::Auto`. A solution-style `tsconfig.json` with a broken sibling reference no longer poisons relative/bare-import resolution for files covered by a healthy sibling. A single `tracing::warn!` per unique error surfaces the broken chain to the user. Path aliases in the broken workspace remain unresolvable (they were already broken in `tsc`). See issue #97.
- **TanStack Router generated route tree imports**: active TanStack Router or Start projects suppress unresolved relative imports ending in `/routeTree.gen`, matching the generated `src/routeTree.gen.ts` artifact commonly imported as `./routeTree.gen` from app routers. The suppression is plugin-gated and does not hide ordinary missing relative imports. See issue #646.
- **TanStack Start virtual runtime modules**: active TanStack Router or Start projects treat `tanstack-start-manifest:` and `tanstack-start-injected-head-scripts:` specifiers as framework virtual module prefixes, covering runtime imports such as `tanstack-start-manifest:v` without hiding ordinary package names that lack the colon suffix. The suppression is plugin-gated and applies to both `unlisted-dependency` and `unresolved-import` reporting through `virtual_module_prefixes()`. See issue #636.
- **rspress `@theme` virtual module**: the `rspress` plugin activates on `rspress` or any `@rspress/`-scoped dependency (e.g. `@rspress/core`) and contributes `@theme/` and `@theme-original/` as `virtual_module_prefixes()`. rspress resolves the bare `@theme` import and `@theme/<component>` subpaths to the active theme layer at build time, so the suppression covers both `unlisted-dependency` and `unresolved-import` for those specifiers. The trailing-slash prefix also covers the bare `@theme` specifier via the exact-bare branch of `matches_virtual_prefix`, while real scoped packages such as `@theme-ui/core` are not swallowed. For Docusaurus projects the existing `@docusaurus` plugin already covers bare `@theme` the same way (no change needed there); the gap was that rspress had no plugin. Content roots (docs `.mdx` reachability) are out of scope. See issue #756.
- **React Router and Remix generated route type imports**: active React Router or Remix projects suppress unresolved type-only relative imports starting exactly with `./+types/`, matching generated route type modules such as `./+types/root`. The suppression is plugin-gated and type-only, so runtime imports under the same prefix still report as unresolved and projects without either plugin keep the normal missing-import finding. See issue #645.
- **SCSS partial resolution**: `@use 'variables'` and `@import '../styles/variables'` resolve to `_variables.scss` per SCSS convention. Extensionless bare specifiers in `.scss`/`.sass` files (excluding `sass:*` built-ins) are normalized to relative paths at extraction time. Resolution fallback tries: (1) `_filename` partial prefix, (2) `specifier/_index` directory index, (3) `specifier/index` plain index. Prevents false unresolved-imports, unlisted-dependencies, and unused-files from SCSS partial references.
- **SCSS include paths**: framework plugins contribute absolute directories to `PluginResult.scss_include_paths`. The Angular plugin reads `projects.*.architect.build.options.stylePreprocessorOptions.includePaths` from `angular.json`; the Nx plugin reads the same key from `project.json`. Nx workspace tokens (`{projectRoot}`, `{workspaceRoot}`) in include paths are expanded before resolution: `{projectRoot}` is computed from the config file's parent relative to the workspace root, `{workspaceRoot}` is stripped (paths are already resolved from workspace root). See issues #103, #114. Bare SCSS specifiers that fail the file-local `try_scss_partial_fallback` retry against each include dir, applying `<name>`, `_<name>`, `<name>/_index`, `<name>/index` candidates. Only triggers for `.scss`/`.sass` importers and normalized bare specifiers (those prefixed with `./` at extraction time) — explicit parent-relative paths like `../shared/vars` are left alone.
- **ng-packagr library entry file**: the Angular plugin reads `lib.entryFile` from `ng-package.json` / `ng-package.prod.json` and credits it as a package entry point, resolved relative to the directory containing the config (via `normalize_config_path`, so `packages/angular/ng-package.json` with `entryFile: "src/public-api.ts"` yields the `packages/angular/src/public-api.ts` entry in a monorepo, prefixed back by the registry). When `lib.entryFile` is omitted (or `lib` absent / the body is malformed), it falls back to ng-packagr's schema default `src/public_api.ts` (underscore, per `ng-package.schema.json`; modern hyphenated `public-api.ts` libraries set `entryFile` explicitly); a non-existent default simply matches nothing. Secondary entry points live in nested `ng-package.json` files in subdirectories that the non-recursive config discovery never reaches, so the primary config's `resolve_config` additionally scans the package subtree (bounded depth, skipping `node_modules` / `dist` / `out` / `tmp` / `coverage` / hidden dirs; same-directory siblings left to discovery) and resolves each nested config's entry file relative to its own directory. The entry file becomes a reachability root, so its re-export chain (`export { useHead } from './composables'`) keeps the public API surface alive instead of reporting it as unused files / unused exports, while files outside that graph stay flagged. The plugin also activates on a `ng-packagr` dependency (in addition to `@angular/core`); enabler detection reads `peerDependencies`, so an ng-packagr library that lists `@angular/core` only as a peer dep still activates. See issue #606.
- **SCSS node_modules resolution**: bare SCSS/Sass specifiers that fail both the local-partial and includePaths fallbacks are retried against `node_modules/` by walking up from the importing file to the filesystem root (matching Node.js / Sass semantics). `try_scss_node_modules_fallback` probes `<nm>/<pkg>/<path>.{scss,sass,css}`, `<nm>/<pkg>/<parent>/_<name>.{scss,sass}` (partial convention), and `<nm>/<pkg>/<path>/{_index,index}.{scss,sass}` (directory index). The CSS-extension append handles Sass's fallback for files like `animate.css/animate.min` → `animate.css/animate.min.css`. Because `node_modules/` files are outside fallow's file index (excluded by default ignore patterns), candidates are probed via `is_file()` on disk and a successful hit returns `ResolveResult::NpmPackage` (extracted via `extract_package_name_from_node_modules_path`), keeping `unused-dependencies` / `unlisted-dependencies` accurate. Scoped packages (`@scope/pkg/path`) are not prefixed with `./` by `normalize_css_import_path`, so they flow through `oxc_resolver`'s bare-module path directly and never reach this fallback. See issue #125.
- **SCSS importer JS/TS rejection**: when the importing file is `.scss` / `.sass` AND the standard resolver returns a path with a JS/TS-family extension (`.tsx`, `.ts`, `.mts`, `.cts`, `.gts`, `.js`, `.jsx`, `.mjs`, `.cjs`, `.gjs`), the result is rejected and resolution re-routes through the SCSS-aware fallback chain (CSS-extension probe, partial convention, include paths, node_modules). When those also fail, the import is reported as `Unresolvable` rather than falling through to the JS/TS extension list. Sass's resolution algorithm only ever considers `.css` / `.scss` / `.sass` files, so a sibling `.tsx` cannot legally satisfy a Sass `@use` / `@import` even though `oxc_resolver`'s extension list places `.tsx` before `.scss`. Without this guard, `@use 'Widget'` from a `.scss` file resolved to `Widget.tsx` whenever both files existed, creating phantom 3-file circular dependencies in standard CSS-modules / Angular `styleUrls` patterns where `.tsx` components import their own `.scss` and a sibling `.scss` shares variables/mixins via `@use`. See issue #245.
- **Entry point source fallback for ignored output dirs**: when `package.json` `main`/`module`/`exports` resolves inside an output directory (`dist`, `build`, `out`, `esm`, `cjs`) and the direct `src/` mirror in `try_output_to_source_path` does not exist, `resolve_entry_path` probes `src/index.*`, `src/main.*`, `index.*`, `main.*` in the package root. Prevents the entire `src/` tree from becoming unreachable in libraries whose compiled entry (e.g. `dist/esm2022/index.js`) has no one-to-one source mirror. Emits a single `tracing::info!` per fallback. See issue #102.
- **Workspace package self-reference & cross-workspace fallback**: when a bare specifier's package name matches the root package or a workspace package and normal resolution either fails or lands inside `node_modules` for that same package, `try_workspace_package_fallback` first honors matching `package.json` `exports` entries and maps project-relative output targets (`dist`, `build`, `out`, `esm`, `cjs`) back to tracked source files. For workspace manifests with no `exports` map, it also tries the stripped package subpath directly against the source layout before trusting legacy `main`, `module`, or `types` fields that may point at missing prebuild output. It then falls back to resolving the stripped subpath (`button`, `internal/base`, etc.) against a synthetic importer inside the package root, using `oxc_resolver`'s `resolve_file` so directory indices, source extensions, and the workspace's own `tsconfig.json` aliases still apply. Package `imports` (`#...`) use only the nearest owning package manifest, and unmatched hash aliases remain unresolved instead of becoming magical global aliases. Package `imports` / `exports` array targets preserve fallback order, and package `imports` targets that point at external packages credit the target dependency. Source-resolved package imports carry dependency usage metadata so static imports, dynamic literal imports, and re-exports keep `unused-dependencies` and `unlisted-dependencies` accurate; unresolved re-export sources also feed `unresolved-import` diagnostics. Handles Node.js v12+ self-referencing (`import '@org/pkg/sub'` from inside `@org/pkg`, common in Angular libraries built with `ng-packagr`), prebuild workspace `exports`, no-exports workspace packages with prebuild legacy fields, and Nitro-style package `imports`. The fallback is also reached when a tsconfig `paths` entry maps a sibling-package specifier at unbuilt output (`"@scope/*": ["../*/dist/index.d.ts"]`, a common pre-build monorepo convention): the TypeScript plugin registers `@scope/` as a path alias, so the import matches `matches_plugin_alias` and `try_path_alias_fallback` runs first, but its dist target does not exist pre-build, so `resolve_specifier` falls through to `try_workspace_package_fallback` for a valid bare package specifier before reporting `unresolved-import`. Without that fall-through every consumer of such a package reported `unresolved-import` plus a false `unused-dependency`. See issues #106, #641, #725, #757, and PR #666.
- **Segment-aware plugin-alias gate**: the `matches_plugin_alias` check in `resolve_specifier` (which routes a specifier into the path-alias branch) now matches a registered alias prefix at a path-segment boundary via `specifier_matches_alias_prefix`, instead of a raw `starts_with(prefix)`. This mirrors `alias_match_remainder` in `fallbacks.rs`: a bare prefix (`@`, `~`) matches only on an exact hit or a `/`-delimited continuation, while a prefix that already ends in `/` matches any continuation. A config registering a bare `"@"` alias (e.g. `"@" -> "src"`) no longer captures every scoped dependency (`@radix-ui/react-checkbox`); those resolve as normal npm packages, while `@/foo` still enters the alias branch. Without the fix, a bare `"@"` key routed every `@scope/*` import into the alias fallback, which failed and cascaded into false `unresolved-import` + `unused-dependency`. See issue #838.
- **Deno import schemes (`jsr:`, `npm:`, URL)**: `resolve_specifier` treats `jsr:<spec>` and URL specifiers (`https://`, `http://`, `data:`) as external files, so Deno / Supabase Edge Function imports are never reported as `unresolved-import` nor misclassified as npm dependencies (previously `jsr:@std/path` collapsed to a bogus `jsr:@std` unlisted dependency). `npm:<pkg>[@version][/subpath]` is normalized by `normalize_npm_specifier` (scheme plus `@version` selector stripped, scope and subpath preserved: `npm:@supabase/supabase-js@2` becomes `@supabase/supabase-js`, `npm:preact@10/hooks` becomes `preact/hooks`) and resolved as an ordinary bare specifier, so a declared dependency is credited; a bare `npm:` with an empty body is treated as external. A package whose every import in a file used the `npm:` scheme is self-declaring (the Supabase / Deno convention of inlining the dependency in the specifier) and is not reported as `unlisted-dependency`, mirroring the per-file `bun:` carve-out (`package_imports_are_all_npm_scheme`); a package also imported via a bare specifier in any file stays reportable. These schemes never appear in Node projects, so non-Deno resolution is unchanged. See issue #624.
- **Playwright `webServer.command` CLI dependencies and script entries**: the Playwright plugin's `resolve_config` reads `webServer.command` from `playwright.config.{ts,js}` in both the object form (`webServer: { command, cwd }`) and the array form (`webServer: [{ command, cwd }, ...]`, via the generic `config_parser::extract_config_array_object_string_pairs` helper; the two forms are mutually exclusive at the AST level so no double-count). Each command is parsed by the shared `scripts::analyze_command`, the same primitive that analyzes package.json scripts, so invoked npm binaries (`srvx --port 3000` credits `srvx`; node runners like `tsx` / `ts-node` are credited as packages, `node` / `bun` excluded) flow into `referenced_dependencies`, and local file arguments (`tsx scripts/e2e-server.ts` seeds `scripts/e2e-server.ts`) are added to `setup_files` so the launched script stays reachable. File arguments resolve relative to the config file's directory (matching Playwright's `webServer.cwd` default and the same base used for `globalSetup` / `globalTeardown`), so a nested `apps/web/playwright.config.ts` discovered from the project root resolves its paths under `apps/web`, not the repo root. A `webServer.cwd` (per object or per array element) overrides that base (resolved relative to the config directory; an absolute cwd replaces it). Binary-to-package resolution still uses the project root since it owns `node_modules`. Commands that delegate to a package manager (`npm run start`, `yarn dev`) credit nothing, since the underlying script's own dependencies are analyzed separately, and template-literal / function-call commands are silently skipped (a syntactic parser cannot recover them). Removes the need for manual `ignoreDependencies` entries for CLI-only packages invoked by the e2e dev server. Existing `globalSetup` / `globalTeardown` handling is unchanged. See issue #621.

## Graph-level
- **Type-only cycle filtering**: `import type` edges carry `is_type_only` through `ImportedSymbol` to cycle detection. Edges where all symbols are type-only are excluded from Tarjan's SCC successor list, preventing false circular dependency reports from type-only bidirectional imports.
- **`export *` chain propagation**: multi-level barrel file chains fully resolved for transitive usage tracking. The fixpoint loop in `crates/graph/src/graph/re_exports/mod.rs::resolve_re_export_chains` terminates naturally via the monotone dedup-by-`from_file` check in `propagate.rs`; a defensive safety backstop bounded by `re_export_info.len() + 1` iterations guards against future regressions. Chain depth is unbounded in practice (the old `max_iterations = 20` cap silently truncated barrels with deeper chains; removed for #442). Re-export cycles are detected up front via iterative Tarjan SCC over the (barrel, source) edge subgraph; cycles surface BOTH as one `tracing::warn!` per cycle naming the member paths (RUST_LOG=warn discoverability, unchanged from #442) AND as a user-visible `re-export-cycle` finding on `AnalysisResults.re_export_cycles` (#515). Self-re-exports (a barrel re-exporting from itself, often a rename leftover) carry `kind: "self-loop"`; multi-node SCCs carry `kind: "multi-node"`. The finding is file-scoped (no per-edge line/col anchor); suppression via `// fallow-ignore-file re-export-cycle` (or aliases `re-export-cycles` / `reexport-cycle` / `reexport-cycles`) on any member file drops the cycle from results, and per-file `overrides.rules.re-export-cycle` is a no-op because the cycle spans multiple files (resolver emits a load-time warn pointing at the working escape hatch). Synthetic bridge stubs created during multi-hop star propagation inherit `is_type_only` from the triggering re-export edge so multi-hop `export type *` chains tag the stub correctly; when both a type-only and a value star edge reach the same source for the same name, a `synthetic_stubs` side set tracks the stub identity so the value edge can downgrade `is_type_only: false`. Real `export type Foo` declarations on the source are never in the side set and stay type-only. Type-only re-export cycles (`export type * from './b'` paired symmetrically) still surface as findings, because chain propagation through the loop is the same no-op as for value cycles. See `decisions/005-re-export-chain-resolution.md`.
- **`export * as Foo from './bar'` namespace re-export propagation**: a barrel that namespace-re-exports a module records a `ReExportEdge { imported_name: "*", exported_name: "Foo" }` plus a synthesised stub `ExportSymbol` named `Foo` on the barrel. The Phase 2 narrowing path only handles `import * as ns` (direct namespace imports), and Phase 4's named-re-export propagation looks for a source export literally named `"*"` (never matches), so a consumer using the named import form (`import { Foo } from './barrel'; Foo.X`) would otherwise leave every export of `./bar` unreferenced. Phase 2c (`graph::namespace_re_exports::propagate_namespace_re_exports`) closes the gap: it iterates re-export edges with `imported_name == "*" && exported_name != "*"`, enumerates the reachable `(barrel_file, exported_name_at_barrel)` pairs via a forward BFS over named re-export edges (so multi-hop chains like `outer_barrel.Foo -> inner_barrel.Foo -> source` match), and for every consumer that imports a reachable name records each `<consumer_local>.<member>` access. Credits are applied with `mark_member_exports_referenced` plus `create_synthetic_exports_for_star_re_exports` on the namespace target file, mirroring `narrow_namespace_references`'s two-step shape so star barrels on the target (`./source/index.ts` doing `export * from './impl'`) get a synthetic stub the Phase 4 chain resolver carries through. Whole-object uses (`Object.values(Foo)`, spread, destructure-with-rest) credit every target export; entry-point barrels exposing the namespace externally also credit every target export, mirroring `propagate_entry_point_star`. Consumer bindings reported as unused (`unused_import_bindings`) are skipped. The walker intentionally does NOT follow `export * as ns from` edges forward at non-seed barrels because the original identifier is hidden behind `ns.<name>` rather than exposed directly. See issue #324.
- **CSS Modules**: default imports (`import styles from '...'`) resolve `styles.className` to named exports via graph-level narrowing. Spread/`Object.values` handled conservatively.
- **Convention auto-import resolution (Nuxt components, scripts, and Pinia stores)**: frameworks that expose components or script bindings by filesystem convention (`<Card001 />` resolving to `components/Card001.vue`, or `useUserStore()` resolving to `stores/user.ts`, with no `import`) are modeled by a first-class `Plugin::auto_imports(&self, root) -> Vec<AutoImportRule>` trait method. The architecture splits across two layers so the per-file extraction cache stays correct: (1) **capture** at extraction. The Vue SFC template scanner (`crates/extract/src/sfc_template/shared.rs::mark_binding_used`) records every PascalCase tag that matches no import and no local binding into `ModuleInfo.auto_import_candidates` (content-local, so it round-trips through the bitcode cache; native lowercase HTML elements and locals are skipped, kebab tags are normalized to their Pascal form). JS/TS and SFC script parsing also records unresolved value-reference identifiers such as `useCounter()`, `formatPrice()`, and `useUserStore()`, including files with zero explicit imports; locally declared, explicitly imported, and type-only names are not captured. (2) **resolve** at graph-build, never cached: `crates/graph/src/resolve/mod.rs::synthesize_auto_import_edges` runs after `apply_specifier_upgrades`, builds a `name -> FileId` table from the active plugins' `auto_imports` rules (mapped via the live file index), skips known JS/Web/Vue/Nuxt built-ins, and pushes a synthetic `ResolvedImport` for each captured name that matches. Name collisions over-credit every match (so `Comments.client.vue` + `Comments.server.vue`, both `<Comments>`, same-name composables in two convention dirs, or same-name stores in two supported dirs, stay reachable). The Nuxt plugin (`crates/core/src/plugins/nuxt.rs`) scans `components/` and `app/components/`, deriving the directory-prefixed PascalCase name with Nuxt's prefix-overlap dedup (`base/foo/Button.vue` -> `BaseFooButton`, `foo/Foo.vue` -> `Foo`, `base/BaseButton.vue` -> `BaseButton`), pre-expanding the implicit `Lazy`-prefixed variant, and stripping `.client` / `.server` / `.global` suffixes so paired files share one name. It scans top-level `composables/`, `app/composables/`, `utils/`, and `app/utils/`, plus recursive `shared/utils/` and `shared/types/`, emitting named-export rules and default-export rules keyed by the derived basename. The Pinia plugin (`crates/core/src/plugins/pinia.rs`) activates on `@pinia/nuxt` and scans direct files in `stores/` and `app/stores/`, emitting named-export rules only for value exports that match `use<Name>Store`; nested stores and custom `storesDirs` remain out of scope. Synthesis is unconditional and additive. The opt-in `autoImports: true` config field gates ONLY Nuxt entry-pattern removal: component fallbacks are dropped unless a root/workspace `nuxt.config` declares `components:`, and composable/util fallbacks are dropped unless it declares `imports:`. Pinia adds no entry-pattern fallback and does not require an extract `CACHE_VERSION` bump because it reuses existing script-level candidate capture. See issues #704, #739, and #740.
- **Nuxt `@nuxt/content` config entry**: `@nuxt/content` v3 reads a root `content.config.ts` (`export default defineContentConfig({...})`) at build time; nothing in app source imports it, so it surfaced as `unused-file` plus `unused-export`. When `@nuxt/content` appears in the nuxt.config `modules:` array, `crates/core/src/plugins/nuxt.rs::resolve_config` credits `content.config.{ts,js,mts,mjs,cts,cjs}` as a default-export entry (entry pattern + `default` used-export rule), resolved relative to the nuxt.config directory via `normalize_config_path` so nested/monorepo configs credit the sibling `content.config` rather than a root one. Gated on module registration, not mere package.json presence: an installed-but-unregistered `@nuxt/content` leaves a genuinely-orphan `content.config` correctly flagged. Scoped to the first-party `@nuxt/content` only; third-party Nuxt module config files are deliberately not hardcoded (declined in #138). No `CACHE_VERSION` bump (`resolve_config` is analysis-time, not extraction). See issue #792.

The tsd plugin activates from `tsd`, marks `.test-d.ts(x)` declaration tests as test entry points, and reads `package.json#tsd.directory` so custom directories such as `test/types` stay reachable.
- **Wrangler config entries follow Wrangler's selected sibling file**: the Wrangler plugin activates from `wrangler`, keeps `wrangler.toml`, `wrangler.json`, and `wrangler.jsonc` config files always used, and reads `main` / `env.*.main` entries only from the highest-precedence sibling config in the same directory. Current Wrangler source selects `wrangler.json`, then `wrangler.jsonc`, then `wrangler.toml`; presence wins over parse success, so lower-precedence `main` values do not become fallback entries. Config files in different directories do not suppress each other, preserving monorepo worker roots. See issue #630.
- **Supabase Edge Functions**: the `supabase` plugin activates on the `supabase` CLI dependency OR a `supabase/config.toml` file OR a `supabase/functions/` directory (the CLI is often installed globally, so the filesystem signals are the robust activation), and marks `supabase/functions/*/index.{ts,tsx,js,jsx,mts,mjs,cts,cjs}` as `EntryPointRole::Runtime` roots so each deployed function entry is not reported as `unused-file` solely because the Node app graph never imports it. Shared code under `supabase/functions/_shared` and per-function helpers stay reachable through their normal relative imports, so a genuinely-orphaned shared file still reports as unused. The `supabase` CLI is credited as a tooling dependency (invoked from scripts, not imported). Pairs with the Deno-scheme resolution handling above so `jsr:` / `npm:` / URL imports inside function files produce no noise. Per-function custom entrypoints declared via `supabase/config.toml` `[functions.<name>] entrypoint` or per-function `deno.json` import maps are out of scope; non-`index` entrypoints fall back to the user's `entry` config. See issue #624.
- **Duplicate export common-importer partition**: a same-name export group is partitioned into connected components by shared importer (union-find: two exports are connected when a third file imports both, or one directly imports the other), and each component is judged independently. Singleton components (an export sharing no importer with any sibling) are dropped as unrelated leaves (e.g., SvelteKit route modules in different directories that coincidentally export the same name). Value/type self-suppression (a lone value paired with a lone type alias is the normal TypeScript runtime-value + type pattern, not a duplicate) is applied per component, so a cross-package member that shares no importer with the rest of the group can no longer inflate a sibling component's value count and defeat its suppression. A finding is emitted only when at least one component survives with two or more connected locations. Previously this was a whole-group boolean (`has_common_importer`): if any pair in the group shared an importer, every member was emitted, which let an unrelated cross-package export defeat the value/type check (issue #848).
- **Duplicate-exports honors `ignoreExports`**: a file matched by a config `ignoreExports` rule is excluded from duplicate-exports grouping (`exports: ["*"]` excludes the whole file, a name list excludes only those names). Built for shadcn / Radix / bits-ui-style component libraries where many `components/ui/<name>/index.ts` barrels intentionally export the same short names (`Root`, `Content`, `Trigger`). Plugin / framework `used_exports` rules are NOT consulted for duplicate-exports gating: they say "this export is framework-managed", not "this export's duplication across barrels is intentional". Per-file `overrides.rules.duplicate-exports` is a no-op because the rule's severity is project-wide and the finding spans multiple files; the resolver emits a load-time `tracing::warn` for any `overrides.rules.duplicate-exports` or `overrides.rules.circular-dependency` setting pointing users at the working escape hatch. See issue #317.
- **Decorated class members**: members with decorators (NestJS `@Get()`, Angular `@Input()`, TypeORM `@Column()`) default to NOT being reported unused, because frameworks consume them reflectively. The `MemberInfo.decorator_names: Vec<String>` field captures the FULL dotted path of each decorator (`@step("x")` -> `"step"`; `@ns.foo` -> `"ns.foo"`). The skip predicate in `find_unused_members` consults `FallowConfig.ignore_decorators`: a method whose every decorator path is in the ignore set is checked normally; a method with any non-ignored decorator stays skipped. Default empty list preserves the legacy skip-all behavior. Matching rule: dotted entries match the full path; bare entries match the leftmost segment, so `"decorators"` collapses every `@decorators.*` decorator. Both `"@step"` and `"step"` accepted at config load (leading `@` stripped). Angular signal-initializer properties (`input()`, `output()`, etc.) carry `has_decorator: true` with empty `decorator_names` and always skip; the gate has no name to match against the ignore list. Unmatched ignore-list entries emit a one-time `tracing::warn!` at end of run via `IgnoreDecoratorSet::warn_unmatched`, mirroring `ClassMemberAllowlist::warn_unmatched_patterns`. The gate operates on the DECLARING class's `decorator_names`; inherited members carry their own (or no) decorators. See issue #471.
- **Entry-point public API class members**: class methods/properties on classes exposed through non-private package entry points are not reported as `unused-class-member` by default. The detector requires public-API provenance from an actual non-private package entry point re-export (`main`, root `exports`, subpath `exports`), from a barrel inside the transitive `export *` closure rooted at one, or from `src/**/index.*` source subpath indexes when the package has no `exports` map. Do not key this on `ReferenceKind::ReExport` alone: internal barrels also create re-export references and must keep member-level detection. Named, renamed, default-as-named, package subpath exports, source-first root `index.js` fallbacks, exportless source subpath indexes, and multi-hop star barrels are covered. The skip is class-member-only; enum member behavior is unchanged, private app packages do not get this skip, and reachable internal classes imported by an entry point but not re-exported still report unused public members. See issue #643.
- **Configurable class member allowlist**: the built-in Angular/React lifecycle check (see `predicates::lifecycle`) is extended at analysis time with two user-controlled sources: (1) top-level `FallowConfig.used_class_members`, (2) `Plugin::used_class_members()` / `ExternalPluginDef.used_class_members` on active plugins. Both sources feed `AggregatedPluginResult.used_class_members`, which `analyze/mod.rs` merges with the config list into a single `FxHashSet<&str>` passed to `find_unused_members`. Intended for third-party interface patterns where a library calls consumer methods reflectively (ag-Grid's `agInit`/`refresh`, Web Components' `connectedCallback`). The allowlist is scoped to class members — matching enum member names are still flagged. Member strings containing `*` or `?` compile as `globset::Glob` matchers, supporting `"*"` (every member on a heritage-matching class), `"enter*"` / `"*Handler"` (prefix / suffix), and `"on*Event"` (combined). Globs run alongside exact-name matches and respect any `extends` / `implements` heritage clause on a scoped rule. Glob patterns matching zero members emit a single `WARN` per pattern at the end of the run via `ClassMemberAllowlist::warn_unmatched_patterns`, surfacing dead allowlist entries; exact-name entries do not warn. See issues #98, #254.
- **Lexical custom-node lifecycle members**: the `lexical` plugin (enabler: `lexical` or any `@lexical/`-scoped package) credits the methods Lexical calls reflectively on custom nodes, so classes that `extends DecoratorNode` / `ElementNode` / `TextNode` do not surface their framework-invoked methods as `unused-class-member`. Implemented via heritage-scoped `UsedClassMemberRule::Scoped` rules in `crates/core/src/plugins/lexical.rs`, mirroring the `lit` / `ember` plugins. The shared lifecycle / serialization / DOM-reconciliation set (`getType`, `clone`, `importJSON`, `importDOM`, `exportJSON`, `exportDOM`, `createDOM`, `updateDOM`, `updateFromJSON`, `getTextContent`) is credited on all three bases; `isInline` is additionally credited on `ElementNode` and `DecoratorNode` only (TextNode is inherently inline and has no `isInline()` hook); `decorate` is credited on `DecoratorNode` only. The per-base layering means a genuinely-dead `isInline` on a TextNode subclass or `decorate` on an ElementNode subclass still surfaces. Method names verified against the Lexical custom-node docs (lexical.dev/docs/concepts/nodes). Heritage matching is on the direct superclass name, so an intermediate base between the user class and the Lexical base is not covered; custom nodes extend a Lexical base directly in practice. Non-lifecycle methods on a node subclass are still reported. No `tooling_dependencies()` is declared because Lexical packages are directly-imported runtime deps. Replaces the manual scoped `usedClassMembers` allowlist users hand-write today. See issue #628.
- **Ionic Angular page lifecycle members**: the `ionic` plugin (enabler: `@ionic/angular`) credits Ionic's Angular page lifecycle methods that the router outlet invokes by name: `ionViewWillEnter`, `ionViewDidEnter`, `ionViewWillLeave`, and `ionViewDidLeave`. Implemented as exact active-plugin `used_class_members()` entries in `crates/core/src/plugins/ionic.rs`, matching Ionic runtime behavior where the methods are called by name with or without the optional TypeScript interfaces (`ViewWillEnter`, `ViewDidEnter`, `ViewWillLeave`, `ViewDidLeave`). Other `ionView*` typos still report as `unused-class-member`. The plugin also keeps `ionic.config.json` reachable and treats `@ionic/cli`, `@ionic/angular-toolkit`, and `ionicons` as Ionic tooling/runtime-template dependencies when the Ionic Angular plugin is active. No `CACHE_VERSION` bump: the names are interpreted at analyze time. See issue #868.
- **NestJS framework-dispatched lifecycle / handler members**: the `nestjs` plugin (enabler: `@nestjs/core`) credits the methods Nest invokes reflectively on consumer classes, so they are not reported as `unused-class-member`. Implemented via heritage-scoped `UsedClassMemberRule::Scoped` rules in `crates/core/src/plugins/nestjs.rs` (one rule per Nest interface, all `implements`-scoped, never `extends`), mirroring the `lit` / `lexical` / `ember` plugins. Handler dispatch methods are scoped to their single interface: `configure` (`NestModule`), `canActivate` (`CanActivate`), `intercept` (`NestInterceptor`), `transform` (`PipeTransform`), `catch` (`ExceptionFilter`), `use` (`NestMiddleware`). The five lifecycle interfaces (`OnModuleInit`, `OnModuleDestroy`, `OnApplicationBootstrap`, `BeforeApplicationShutdown`, `OnApplicationShutdown`) each credit ALL five lifecycle method names (`onModuleInit`, `onModuleDestroy`, `onApplicationBootstrap`, `beforeApplicationShutdown`, `onApplicationShutdown`) rather than only the one method the implemented interface declares. This is intentional and more correct: Nest's lifecycle manager dispatches by duck-typed method presence, so a class that implements one lifecycle interface but also defines a sibling hook method has that sibling invoked at runtime too. Heritage matching is on `implements`, so an ordinary class that implements none of these interfaces still surfaces a genuinely-unused `transform` / `configure` / lifecycle method, and non-lifecycle methods on a Nest class are still reported. No `CACHE_VERSION` bump: the rules are interpreted at analyze time against existing `implements` heritage data and never enter the extraction cache. See issue #843.
- **Native Error subclass `name` members**: a class member named `name` declared on a class whose heritage reaches a native ECMAScript error constructor (`Error`, `TypeError`, `RangeError`, `SyntaxError`, `ReferenceError`, `EvalError`, `URIError`, `AggregateError`) is treated as runtime-used and never reported as `unused-class-member`. Custom error classes set `name` so logs, serializers, and `err.name === "..."` discrimination expose the error type, none of which is a static member access fallow can see. Implemented as a built-in heritage-scoped suppression in `crates/core/src/analyze/unused_members.rs` (`is_error_subclass_runtime_member`), mirroring the existing native-Custom-Elements built-in (`is_native_custom_element_lifecycle_method`, `super_class == Some("HTMLElement")`); no plugin or dependency gate because the bases are JavaScript globals. `build_error_subclass_export_keys` seeds on classes that directly extend a native error name, then walks `extends` chains downward via `build_parent_to_children`, so a transitive `class ApiError extends DomainError` where `DomainError extends Error` (resolved through imports, local exports, and re-export origins) is also covered. The rule is heritage-scoped: an ordinary `class Person { name = "x" }` still reports an unused `name`, and members other than `name` on error subclasses (`message` / `stack` / `cause` are inherited and rarely re-declared; arbitrary helper methods) still report. Seeding is by direct super-class name, matching the HTMLElement built-in, so a local class shadowing a native error name is treated as a native error base (pathological in practice). Same-file NON-exported intermediate base classes are not resolved, the same limitation `build_parent_to_children` carries for all heritage-aware member crediting. No `CACHE_VERSION` bump: only analyze-time interpretation of the existing `class_heritage` / `super_class` extraction data changes. See issue #620.
- **Cross-package namespace re-export through object literal**: `import * as foo from './bar'; export const API = { foo }` in a barrel followed by `import { API } from '@scope/lib'; API.foo.bar` in another package credits `bar` on `./bar.ts`. The visitor records per-module `NamespaceObjectAlias` entries `{ via_export_name, suffix, namespace_local }` at finalize time (covering shorthand `{ foo }` and nested forms `{ motionNet: { adEngine } }`); both `into_module_info` and the SFC `merge_into` paths populate them. The graph layer threads them onto `ResolvedModule` and a Phase 2b pass (after reference population, before reachability) walks each consumer's `member_accesses` for `object == "<consumer_local>.<suffix>"`, looks up the namespace import's target file via the module's resolved imports filtered to `ImportedName::Namespace`, and runs the `mark_member_exports_referenced` + `create_synthetic_exports_for_star_re_exports` pair from `narrowing.rs` so star-barrel namespace targets (`./foo/index.ts` doing `export * from './bar'`) get a synthetic stub export the Phase 4 chain resolver can carry to the real defining file. Default-export aliases match against `ImportedName::Default`. **Consumer-side multi-hop** (issue #310): the alias-defining file may sit behind 1-3 named-re-export barrels (`pkg/index.ts` -> `pkg/methods/index.ts` -> `pkg/methods/methods.ts`); the matcher does a forward BFS over re-export edges starting from `(alias_file, via_export_name)` to enumerate every `(barrel_file, exported_name_at_barrel)` pair through which the alias is reachable, then matches consumer imports against the full set. Plain `export *` propagates names unchanged; `export { A as B }` propagates the renamed `B`; `export * as ns` is skipped at non-seed barrels (its source identifiers are hidden behind `ns.<name>`, not exposed directly). **Target-side multi-hop** (issue #328): the namespace target itself may chain through `export * as N from './source'` namespace re-exports (`barrel.ts` does `export * as inner from './leaf'`; consumer accesses `API.foo.inner.X`). After the initial Phase 2b credit lands on a name namespace-re-exported on the target, the matcher walks chained `export * as <name> from './source'` edges on the alias-target side and propagates the consumer's deeper accesses (`<expected_object>.<member>.<X>`) onto the underlying source recursively via `ChainWalkCtx`. Cycle-protected by a per-access visited set keyed on module index. Bumps the extract `CACHE_VERSION` because the new `namespace_object_aliases` field changes serialized module state. See issue #303 (cross-package case) building on issue #269 (same-file case which `resolve_bound_member_accesses` already handled by re-emitting the access against the namespace local); #310 (consumer-side multi-hop); #328 (target-side namespace-re-export chain).
- **Cross-package member access through barrels**: `import { Foo } from '@scope/lib'` followed by `Foo.bar` correctly credits `bar` on the originating file even when `@scope/lib`'s entry resolves to a barrel that re-exports `Foo` from a sibling module. Phase 4 chain resolution synthesizes a stub `ExportSymbol` on the barrel for reference tracking, so without further propagation the access map keys at the barrel and the detection loop's origin-keyed lookup misses where `members` are populated. `find_unused_members` walks each access key through `ReExportEdge` chains (named re-exports, renamed re-exports, and `export *` fan-out when no named edge matches) to every defining-site export and copies the access set. Cycle-protected. `export * as ns from './mod'` edges are intentionally skipped because two-level `ns.Foo.member` access isn't tracked at extract time. Same propagation runs for `whole_object_used_exports` (`Object.values(Foo)` on a re-exported enum). Covers both `unused_enum_members` and `unused_class_members`. See issue #178.
- **Inheritance-aware class members**: `extends` clauses are extracted to build an inheritance map. Parent class `this.*` accesses propagate to child class overrides (e.g., `BaseShape.describe()` calling `this.getArea()` credits `Circle.getArea()`). External member accesses propagate bidirectionally through the hierarchy. `super.method()` calls inside a subclass body are attributed to the parent via a `class_super_stack` in the visitor: the super-class identifier from the `extends` clause is pushed on class entry and popped on exit, and `Super.member` expressions record a `MemberAccess { object: <super_local>, member }` which `local_to_imported` then resolves to the parent's export. This covers the common pattern of a base class method only called via `super.method()` from subclasses. Only simple identifier extends are tracked (`class Foo extends Bar`), not mixin patterns. Methods called through parent-typed parameters remain unattributed (requires type inference, out of scope per ADR-001). See issue #130.
- **Private type leaks**: an exported value, function, or class whose public signature references a same-file type that is NOT exported by name is reported as `private-type-leak`. Default severity: `Off` (opt-in API hygiene check). Enable with the `--private-type-leaks` filter for a one-off run, or `private-type-leaks: "warn"` / `"error"` in `rules` for persistent enforcement. The CLI filter promotes the rule from `Off` to `Warn` for that invocation only via `IssueFilters::activate_explicit_opt_ins` in `crates/cli/src/check/mod.rs`. Detection runs in `find_private_type_leaks` (`crates/core/src/analyze/unused_exports.rs`) over `module_info.public_signature_type_references` (collected by the visitor's `SignatureTypeCollector`) cross-checked against `module_info.local_type_declarations`. Same-file private = the referenced name is in `local_type_declarations` AND not in the module's `exports`. References from ECMAScript private fields (`#field: T`) and TypeScript `private` accessibility members are NOT collected (the visitor short-circuits in `collect_class_signature_refs`), so they cannot leak. Qualified type names (`Foo.Bar.Baz`) reduce to the root identifier (`Foo`), since the leak is "the root namespace is private", not "every dotted segment is". Storybook story files (`*.stories.{ts,tsx,js,jsx,mts,cts,mjs,cjs}` and `*.story.*` variants) are skipped via `is_storybook_file`: the canonical `type Story = StoryObj<typeof X>; export const Default: Story = ...` pattern would otherwise generate one leak per story across thousands of UI library files. Framework routing convention files are also skipped via `is_route_convention_file` (Next.js App Router page/layout/template/loading/error/not-found/route/default/global-error/forbidden/unauthorized + the metadata files opengraph-image/twitter-image/icon/apple-icon/manifest/sitemap/robots; Next.js Pages Router; Gatsby `templates/`; Remix v2 + TanStack Router top-level + folder-route entries route/_layout/_index/index/__root; Expo Router `_layout` and `+*` files). All patterns prefixed with `**/` so they match in monorepo subpackages (`packages/web/src/app/.../page.tsx`). Real-world delta: amictus.ai 62 → 59 leaks (3 Next.js convention noise eliminated, 59 remaining are real React `Props`-pattern findings that warrant exporting the Props type). Companion logic in `suppress_signature_backing_types` removes any `unused_types` finding whose export is the type name backing a public signature in the same file, so exporting a backing type to fix one leak does not create a new "unused type" finding.
- **JSDoc visibility tags** (`@public`, `@internal`, `@beta`, `@alpha`, `@expected-unused`): exports annotated with any of these JSDoc/TSDoc tags are never reported as unused. Stored as a `VisibilityTag` enum on `ExportInfo` and `ExportSymbol`, preserving the semantic distinction (Public = API surface, Internal = sister packages/build tools, Alpha = early preview, Beta = unstable, ExpectedUnused = intentionally unused). Priority when multiple tags present: Public > Internal > Alpha > Beta > ExpectedUnused. Only `/** */` block JSDoc comments recognized. `@api public` TSDoc variant also supported. Word-boundary checked (e.g., `@internalizer` does not match). Survives incremental cache roundtrip via `u8` discriminant encoding. `@expected-unused` differs from other tags: when the tagged export IS actually used (has references from reachable modules), a stale suppression is reported instead of silently accepting it. This supports the pattern of marking API-completeness exports that become adopted over time.
- **Stale suppression detection**: after all detectors run, `SuppressionContext` compares consumed vs. all suppression comments. Unmatched `// fallow-ignore-next-line` and `// fallow-ignore-file` comments are reported as stale. Default severity: `Warn`. - **Script multiplexer handling**: `concurrently`, `npm-run-all`, `npm-run-all2`, `run-s`, `run-p`, `run-s2`, `run-p2` are recognized as script multiplexers. This prevents false "unused dependency" reports for packages referenced only via `npm:` prefixed script delegation (e.g., `concurrently "npm:dev" "npm:test"`).
- **Entry export validation** (`--include-entry-exports`): when enabled, entry file exports (package.json main/module/exports, framework convention files) are subject to unused-export detection instead of being automatically marked as used. Catches typos in framework exports (e.g., `meatdata` instead of `metadata` in Next.js pages). Still respects plugin `used_exports` rules for partial checking. CLI-only flag, not in config schema.
- **Single-file filtering** (`--file`): scopes check output to specified file(s). Full graph still builds (correct results require cross-file analysis), but only issues in matching files are reported. Dependency-level issues (unused deps, dev deps, type-only deps) are suppressed in file mode since they are project-wide properties. Warns when a path is not found. Useful for lint-staged pre-commit hooks.
- **Hotspot ownership signals** (`--ownership` on `health --hotspots`): per-file ownership metrics derived from `git log --use-mailmap --format='%at|%ae'`. Computed only when `--hotspots` runs and a file passes the `min_commits` threshold. Signals: bus factor (Avelino truck factor — minimum contributors covering ≥50% of recency-weighted commits), contributor count (after bot-pattern filtering), top contributor with stale-days, recent contributors (top-3), declared CODEOWNERS owner, `ownership_state` (`active`, `unowned`, `declared_inactive`, `drifting`), drift (file age ≥30d AND original author share <10%), tristate `unowned` (true=no rule matches; false=rule matches; null=no CODEOWNERS file). Declared owners suppress vague git-history drift only when fallow can match them to an active contributor offline. Emitted via `OwnershipMetrics` on `HotspotEntry` (`skip_serializing_if = "Option::is_none"` for backwards compat). JSON injects ownership-aware actions: `low-bus-factor`, `unowned-hotspot` (with synthesized CODEOWNERS pattern), `ownership-drift` only for `ownership_state == "drifting"`. Email privacy mode (`ownership.emailMode` config or `--ownership-emails` CLI): `handle` (default, local-part only with GitHub noreply unwrap and deterministic same-handle disambiguation), `anonymized` (`xxh3:` non-cryptographic pseudonym), legacy `hash` (same as `anonymized`), `raw`. Bot patterns are globset matches against the raw author email; defaults cover `*\[bot\]*` (escaped brackets), `dependabot*`, `renovate*`, `github-actions*`, `svc-*`, `*-service-account*`. `*noreply*` is intentionally NOT a default because most human GitHub contributors commit from `<id>+<handle>@users.noreply.github.com` (GitHub's privacy default); the actual bot accounts already match via `\[bot\]`. Author emails are interned in `ChurnResult.author_pool` and per-file maps reference them by `u32` index to keep the bitcode cache compact. Cache version bumped to v2.
- **Feature flag detection**: AST-based detection of feature flag patterns. Three categories: (1) environment variables (`process.env.FEATURE_*` with 12 built-in prefixes), (2) SDK calls (40+ built-in patterns for LaunchDarkly, Statsig, Unleash, GrowthBook, Split, PostHog, Vercel Flags, ConfigCat, Flagsmith, Optimizely, Eppo), (3) config objects (opt-in heuristic for objects named "feature", "flag", "toggle"). Reports flag locations, detection confidence (high/low), guard spans. Cross-references with dead code: flags guarding unused exports are highlighted. Configured via `flags` section in config. Default severity: `off`. CLI: `fallow flags`. Inline suppression: `// fallow-ignore-next-line feature-flag` (line-level) and `// fallow-ignore-file feature-flag` (file-level). The suppression check runs in `crates/cli/src/flags.rs` during both built-in and custom flag collection loops, using `is_suppressed()` / `is_file_suppressed()` with `IssueKind::FeatureFlag`. The file-level check is hoisted outside the per-flag loop for efficiency.
- **Client-server-leak security candidates** (opt-in, `fallow security` only): a graph-structural rule that flags a `"use client"` file (React Server Components / Next.js client boundary) whose transitive static-import cone reaches a module that reads a NON-PUBLIC `process.env` secret, plus the direct case where the `"use client"` file reads such a secret itself. Findings are CANDIDATES for downstream agent verification, NOT verified vulnerabilities; there is deliberately no `confidence` / `signal_strength` field, only the structural import-hop `trace`. Implemented in `crates/core/src/analyze/security/mod.rs`: file-level directive capture (`ModuleInfo.directives`, captured by `visit_program` in the extract visitor; `CACHE_VERSION` bumped 104 to 105) identifies client boundaries; secret sources are detected for free from the existing `member_accesses` (`process.env.X` surfaces as `MemberAccess { object: "process.env", member: "X" }`), excluding public-by-convention prefixes (`NODE_ENV`, `NEXT_PUBLIC_`, `VITE_`, `NUXT_PUBLIC_`, `REACT_APP_`, `PUBLIC_`, `GATSBY_`, `EXPO_PUBLIC_`, `STORYBOOK_`) which frameworks inline into the client bundle; a BFS modeled on `reachability::collect_reachable` walks the cone, skipping type-only edges (erased at build, cannot carry a secret). The default rule severity is `off` (`RulesConfig.security_client_server_leak`); `fallow security` forces it to `warn`. The finding lives on `AnalysisResults.security_findings` (`#[serde(skip)]`, excluded from `total_issues`), so it NEVER appears under bare `fallow` or the `audit` gate; the `fallow security` command is the only surface and emits its own `SecurityOutput` envelope (human / JSON / SARIF, SARIF at `level: note` with no CWE). Blind spots are surfaced in-band, never silently dropped: dynamic `import()` patterns the BFS cannot follow bump `security_unresolved_edge_files` (a counted human line: an empty result with a non-zero count is not a clean bill), and `import.meta.env` (the Vite secret convention) is not modeled. Inline suppression: `// fallow-ignore-file security-client-server-leak` (file-level), routed through `SuppressionContext::is_file_suppressed` so the marker is recorded as consumed and not later flagged stale; the rule's default-`off` severity also keeps the suppression from being stale under bare `fallow`. Each finding carries an `actions` array with that file-level suppress hint (`auto_fixable: false`; verification is the agent's job). `--changed-since` / `--workspace` retain `security_findings` by anchor path (core `filter_results_by_changed_files`, cli `filter_to_workspaces`). ADR-021 amendment (2026-05-30) admits this as a cluster-4 graph-structural candidate; external CVE/OSV/SCA/SBOM stay out of scope.
- **Data-driven security candidate catalogue** (opt-in, `fallow security` only): a category-blind, data-driven extension of the security-candidate surface. Where `client-server-leak` is a bespoke graph-reachability rule, this is a syntactic SINK catalogue: the extract layer captures every non-literal call / member-call / member-assign / tagged-template / jsx-attr sink site into `ModuleInfo.security_sinks` (a `SinkSite { sink_shape, callee_path, arg_index, arg_is_non_literal, arg_kind, span }`, where `arg_kind` discriminates the argument shape (`concat` / `template-with-subst` / `object` / `call` / `other`) so a matcher can require the genuinely-unsafe shapes; knowing nothing about CWE categories; `CACHE_VERSION` bumped to 109), and the analyze-layer `tainted_sink` detector (`crates/core/src/analyze/security/tainted_sink.rs`) matches those sites against a catalogue. Findings are CANDIDATES for downstream agent verification, NOT verified vulnerabilities; like `client-server-leak` there is deliberately no `confidence` / `signal_strength` field, only the sink-site `trace` and the catalogue's evidence string. All catalogue findings carry `SecurityFindingKind::TaintedSink` (ONE kind covers every category) with `category` (the catalogue id, e.g. `"dangerous-html"`) and `cwe` (the catalogue's declared CWE number, never fabricated); `client-server-leak` carries neither. The catalogue is a single data file `crates/core/data/security_matchers.toml`, embedded via `include_str!` and parsed once behind a `OnceLock` (`crates/core/src/analyze/security/catalogue.rs`), mirroring the `tooling.toml` precedent: adding a CWE category is ONE `[[matcher]]` edit with ZERO Rust enum/discriminant churn (so long as it maps to an existing `sink_shape` plus a positional non-literal arg). Seeded categories: `dangerous-html` (CWE-79), `command-injection` (CWE-78), `code-injection` (CWE-94, `eval` + `vm.runInNewContext`), `sql-injection` (CWE-89; tightened per the false-negative principle: only a string-concat or interpolated template literal passed to `.query()` / `.execute()` (`arg_kinds = ["concat", "template-with-subst"]`) plus `sql.raw(...)` fire. A bare parameterized `` sql`${x}` `` tagged template and the parameterized object form `.execute({ sql, args })` (arg_kind `object`) deliberately do NOT fire, since both bind safely; this drops the bulk of false positives on Drizzle / postgres.js / slonik apps), `ssrf` (CWE-918), `path-traversal` (CWE-22), `open-redirect` (CWE-601), `weak-crypto` (CWE-327, retitled "Runtime-selectable crypto algorithm" so it does not over-claim), and `unsafe-deserialization` (CWE-502, `js-yaml` + `node-serialize`). The conservative trigger is the only thing that ever fires a candidate: a FULLY-LITERAL argument is never captured at the extract layer, so `el.innerHTML = "<b>x</b>"`, `child_process.exec("ls")`, and `crypto.createHash("sha256")` produce nothing; the catalogue prefers false-negatives over false-positives throughout. `callee_patterns` are segment-aware (not substring): `fetch` matches `fetch` not `myfetch`, and a leading `*.` matches any object (`*.innerHTML` matches `el.innerHTML`). Some matchers are provenance-gated via `import_provenance` (`command-injection` to `node:child_process`, `code-injection`'s `vm` row to `node:vm`, `path-traversal` to `node:path`, `weak-crypto` to `node:crypto`, `unsafe-deserialization` to `js-yaml` / `node-serialize`): the module must import the source, and `command-injection` additionally binding-traces (the import's `local_name` must be the leading identifier of the callee path), mirroring the `child_process.fork()` provenance precedent (`is_node_module_register`-style gating). The default rule severity is `off` (`RulesConfig.security_sink`, kebab `security-sink`, discriminant 28); `fallow security` is the ONLY surface that flips it to `warn` (alongside `security_client_server_leak`), so it NEVER appears under bare `fallow` or the `audit` gate; the finding lives on `AnalysisResults.security_findings` (`#[serde(skip)]`, off `total_issues`). `security.categories` (include / exclude lists on `SecurityConfig`, both unset admits every category) scopes which catalogue ids are active. Candidates anchored in low-value noise files are dropped from generation, matching production-mode dead-code exclusion: `is_low_value_anchor` skips test / spec / story / fixture files (the `PRODUCTION_EXCLUDE_PATTERNS` globset, `literal_separator(true)`, matched on the forward-slash-normalized workspace-relative path) and tooling config files (`is_config_file`, e.g. `vite.config.ts`, `jest.config.js`), so a non-literal `path.join` in a build config or a `.test.ts` is not surfaced. Each `tainted-sink` finding carries a single `TraceHopRole::Sink` ("sink site") trace hop, distinct from `client-server-leak`'s `ClientBoundary` / `Intermediate` / `SecretSource` roles. Blind spots are surfaced in-band, never silently dropped: sink-shaped nodes whose callee cannot be flattened to a static path (dynamic dispatch, computed members, aliased bindings) bump `ModuleInfo.security_sinks_skipped`, summed into the `SecurityOutput.unresolved_callee_sites` counter (a counted human line: a zero finding count with a non-zero value here is NOT a clean bill). Inline suppression: ONE token `// fallow-ignore-file security-sink` covers all categories (file-level), routed through `SuppressionContext::is_file_suppressed` so the marker is recorded as consumed and not later flagged stale; the default-`off` severity also keeps the suppression valid (never stale) under bare `fallow`. Each finding carries an `actions` array with that file-level suppress hint (`auto_fixable: false`; verification is the agent's job, not fallow's). ADR-021 amendment (2026-05-30) admits this as a cluster-4 local code-level candidate layer; external CVE/OSV/SCA/SBOM stay out of scope.
- **File encoding contract for fix + extract** (issue #475): fallow's contract is "UTF-8 with or without BOM; line offsets are computed against the post-BOM view; the BOM, if present on input, is preserved on output by `fallow fix`." `crates/extract/src/lib.rs` strips the leading UTF-8 BOM at every file-read entry point (`parse_single_file_cached`, `parse_single_file`, `parse_from_content`) before computing `content_hash` (xxh3 over post-strip bytes) and before passing source to `parse_source_to_module`. `parse_source_to_module` carries a defense-in-depth `strip_bom` call for out-of-tree callers (fuzzers, integration fixtures, future embedders). `crates/cli/src/fix/io.rs::read_source` returns `Result<Option<(String, EncodingMetadata)>, EncodingError>` where `EncodingMetadata { line_ending, had_bom }` captures the round-trip shape; `stage_fixed_content` re-prepends the BOM bytes (`EF BB BF`) on write when `had_bom`. Mixed CRLF/LF files (both `\r\n` and bare-LF lines present) are rejected with `EncodingError::MixedLineEndings`; the orchestrator translates this into `SkipReason::MixedLineEndings`, surfaces an additive `skipped_mixed_line_endings: M` envelope counter, emits a stderr line naming `dos2unix` and `git config core.autocrlf input` as remediation, and exits with code 2. The skip is NOT self-healing: re-running fallow alone does not normalize the file. Cross-fixer composition preserves the `had_bom` flag because the staged-content fast path re-runs `classify_source` on the staged bytes (which already include the re-prepended BOM from the first fixer). The extraction cache (`CACHE_VERSION` bumped from 88 to 89) invalidates automatically on upgrade because pre-fix caches keyed hashes over BOM-included bytes. The `fix_preview` and `fix_apply` MCP tool descriptions flag the non-self-healing nature for AI agents.

- **Unused pnpm catalog entries**: `pnpm-workspace.yaml`'s `catalog:` map (default catalog) and `catalogs.<name>:` maps (named catalogs) are parsed via `serde_yml` plus a section-aware line scanner (`crates/config/src/workspace/pnpm_catalog.rs`) that records the 1-based source line for every entry. Workspace `package.json` files are walked for dependency values matching the `catalog:` protocol: a bare `"react": "catalog:"` and an explicit `"react": "catalog:default"` both resolve to the default catalog per pnpm's spec; named references use the catalog name verbatim. The detector (`crates/core/src/analyze/unused_catalog.rs::find_unused_catalog_entries`) emits one `UnusedCatalogEntry` per catalog entry no workspace `package.json` references via `catalog:` for the matching catalog group. Each finding also carries `hardcoded_consumers: Vec<PathBuf>`, the workspace `package.json` paths that declare the same package with a non-`catalog:` version range, so consumers can decide whether to delete the catalog entry or switch the consumer to `catalog:` first (the differentiator vs knip's catalog detection). Output sort is `(path, catalog_name, entry_name)` with the default catalog sorting first via `catalog_sort_key`. Default severity `warn` (not `error`) because catalog entries ship zero bytes; matches the rest of the `unused_dev_dependencies` / `unused_optional_dependencies` family. JSON `actions[]` emits a YAML-style `# fallow-ignore-next-line unused-catalog-entry` suppression action via the new `SuppressKind::YamlComment`. Filter-to-workspaces clears catalog findings (whole-project concern, not per-workspace). **Auto-fix** (`crates/cli/src/fix/catalog.rs`, issues #335 and #360): `fallow fix` removes the entry by line-aware deletion (no YAML round-trip writer in the workspace; a full reprint via `serde_yaml_ng` would obliterate comments). Object-form entries (`react:\n    specifier: ^18.2.0`) are detected by consuming subsequent lines whose indent is strictly greater than the entry's own; blank lines stop the consumption to preserve inter-entry whitespace. `fix.catalog.deletePrecedingComments` controls contiguous same-indent YAML comment blocks immediately above removed entries: `auto` (default) deletes blocks that directly follow the parent catalog header or a blank separator, `always` deletes any adjacent block, and `never` preserves leading comments. Two escape hatches keep curated comments safe regardless of policy: any line in the block bearing the `# fallow-keep` marker bails out of deletion (mirrors the `fallow-ignore` inline-suppression convention), and the `auto` policy additionally preserves section-banner blocks whose comment body starts with three or more `=`, `-`, `*`, `_`, `~`, `+`, or `#` characters (`# === React 18 production pins ===`). The JSON `remove_catalog_entry` action carries both `line` (first deleted line, the comment when the policy absorbs one) and `entry_line` (the entry's original 1-based line) so CI annotators and dedup caches keep a stable anchor across policy changes; the human stderr summary appends `(+M catalog comment lines)` after the fixed-issue count when at least one preceding comment line was absorbed. Entries with non-empty `hardcoded_consumers` are skipped (a record with `applied: false`, `skipped: true`, `skip_reason: "hardcoded_consumers"` is emitted so the user knows to migrate the consumer first). Multi-document YAML files (`---` separator) are rejected up front with `skip_reason: "multi_document_yaml"`. When removing the last entry of a catalog group (default `catalog:` or a named `catalogs.<name>:`) leaves the header with no children, the fix rewrites the header to `catalog: {}` / `<name>: {}` instead of leaving bare `key:`, because bare-key parses as null in YAML and pnpm rejects null-valued catalogs with `Cannot convert undefined or null to object` at install time (verified against pnpm 10.33.4). The post-edit content is reparsed with `serde_yaml_ng` as a syntactic sanity gate before persisting. The per-instance `auto_fixable` bool in the JSON `actions[]` flips to `false` for entries with hardcoded consumers (computed in `build_actions` in `crates/cli/src/report/json.rs`, mirroring the existing per-instance pattern used by `unresolved_catalog_references`); agents that filter on the bool no longer apply destructive ops to entries needing consumer-side migration first. After a successful fix that touched `pnpm-workspace.yaml`, the human stderr emits a one-line `Run \`pnpm install\` to refresh pnpm-lock.yaml` reminder. The LSP code action (`crates/lsp/src/code_actions/quick_fix.rs::build_remove_catalog_entry_actions`) mirrors the same guard and uses anchored key-prefix matching to reject substring collisions between sibling catalog entries with shared prefixes (`react` vs `react-native`), but intentionally does not delete leading comments so editor quick fixes stay conservative. See issues #329 (detection), #335 (auto-fix), and #360 (leading comments). See also: the `unresolved-catalog-reference` inverse rule (consumer references a catalog that does not declare the package).
- **Empty pnpm catalog groups**: named `catalogs.<name>:` groups in `pnpm-workspace.yaml` that contain no package entries are detected by `crates/core/src/analyze/unused_catalog.rs::find_empty_catalog_groups`. The top-level `catalog:` map is intentionally ignored even when empty, because it is the default-catalog hook and can be a harmless placeholder. Findings are project-root-relative to `pnpm-workspace.yaml`, sorted by `(path, catalog_name)`, default to `warn`, support YAML line suppressions via `# fallow-ignore-next-line empty-catalog-group`, and are auto-fixable by removing the empty named group header without reprinting the whole YAML document. Changed-file and audit attribution paths must account for the relative manifest path.
- **Unresolved pnpm catalog references**: the inverse of `unused-catalog-entries`. The same YAML parse + consumer walk (`find_unresolved_catalog_references` in `crates/core/src/analyze/unused_catalog.rs`) iterates over every `catalog:` / `catalog:<name>` reference collected in `CatalogConsumers.referenced_with_locations` and emits an `UnresolvedCatalogReference` for any `(catalog_name, package_name)` pair the YAML does not declare. The consumer-side line number is captured by a section-aware `scan_dep_lines` pass over the raw `package.json` source that tracks brace depth so nested-object keys (`peerDependenciesMeta.<pkg>.optional`) cannot be misread as direct dep entries. Each finding carries `available_in_catalogs: Vec<String>`: other catalogs in the same workspace that DO declare the package, sorted lexicographically; agents flip the reference to one of those when non-empty, otherwise add the missing entry to the named catalog. Default severity `error` (matches `unresolved-imports`): `pnpm install` errors with `ERR_PNPM_CATALOG_ENTRY_NOT_FOUND_FOR_CATALOG_PROTOCOL` so the static finding catches the same failure pre-install. Default-catalog rendering uses a special-case "default catalog" phrasing in human / SARIF / LSP / CodeClimate output instead of `catalog 'default'` because users who write bare `catalog:` think of it as "the catalog", not as a named one. Suppression is config-only via the new `IgnoreCatalogReferenceRule { package, catalog?, consumer? }` array on `FallowConfig` (consumer is a glob, all three fields AND together when set); `package.json` does not support comments so inline suppression is structurally impossible. JSON `actions[]` is discriminated by the `available_in_catalogs` shape: `update-catalog-reference` (primary when alternatives exist), `add-catalog-entry` (primary when none do), `remove-catalog-reference` (fallback from the generic spec), plus an `add-to-config` for `ignoreCatalogReferences` with a paste-ready value scoped to the consumer. Filter-to-workspaces retains findings under the active set because each finding is anchored at a consumer `package.json` (unlike unused-catalog-entries which lives in the root YAML). The LSP diagnostic mirrors the catalog-entries pattern with `root.join(&finding.path)` so the URI built from the project-relative path is absolute. See issue #334.
- **Unused pnpm dependency overrides**: pnpm's `overrides:` section (top-level in `pnpm-workspace.yaml` for pnpm 9+, or legacy `pnpm.overrides` in the root `package.json`) lets a workspace force a specific transitive dependency version (typically for a CVE patch). The detector (`crates/core/src/analyze/unused_overrides.rs::find_unused_dependency_overrides`) parses both source forms via `crates/config/src/workspace/pnpm_overrides.rs`, walks every workspace `package.json` to compute the union of declared package names across `dependencies`, `devDependencies`, `peerDependencies`, and `optionalDependencies`, and ALSO parses `pnpm-lock.yaml` to collect every package name appearing in `packages:` / `snapshots:` keys plus any nested `dependencies` / `optionalDependencies` / `devDependencies` / `peerDependencies` map. An `UnusedDependencyOverride` is emitted only for overrides whose target package is absent from BOTH the declared set AND the lockfile-resolved set. Parent-chain shapes (`react>react-dom: ^17`) are USED when EITHER the parent OR the target is declared OR resolved in the lockfile, covering the CVE-fix pattern where the override forces a transitive version that only appears in `pnpm-lock.yaml`. When the lockfile is missing or malformed (`serde_yaml_ng::from_str` fails), the detector degrades to the package.json-only fallback and the `hint` field reminds consumers to verify the override before removal. Lockfile keys are normalized via `package_name_from_lock_key`, which strips the leading `/` used by pnpm 6 keys, the version selector at `@`, and pnpm peer suffixes in parentheses, correctly handling scoped packages (`@scope/pkg@1.0.0` → `@scope/pkg`). Default severity `warn`. Each finding decomposes the raw key into `target_package`, `parent_package` (Option), `version_constraint` (Option, e.g. `Some("<18")` for `@types/react@<18`), and `version_range` (the forced version). Suppression is config-only via `ignoreDependencyOverrides: [{ package, source? }]` on `FallowConfig`: `package` matches the override's `target_package` exactly, optional `source` scopes to `"pnpm-workspace.yaml"` or `"package.json"`. JSON `actions[]` emits `remove-dependency-override` as primary plus an `add-to-config` for `ignoreDependencyOverrides`. Filter-to-workspaces clears findings (whole-project concern). See also: `misconfigured-dependency-overrides` (the entry-shape variant). See issues #336 (detector) and #371 (lockfile-aware check).
- **Misconfigured pnpm dependency overrides**: same parser as `unused-dependency-overrides`, different gate. `find_misconfigured_dependency_overrides` emits `MisconfiguredDependencyOverride` for entries whose key cannot be parsed (`DependencyOverrideMisconfigReason::UnparsableKey`: empty key, dangling separators like `react>`, malformed selectors like `@types/react@<<18`) or whose value is missing/empty (`EmptyValue`). pnpm itself refuses to honor these at install time. The pnpm idiom values `-` (delete), `$ref` (self-reference), and `npm:alias@^1` (npm alias) are explicitly allowlisted and never flagged. Default severity `error`: a malformed override breaks `pnpm install` for every contributor. Findings carry the raw key, raw value, the reason discriminant (serialized as kebab-case `unparsable-key` / `empty-value`), source, path, and line. Suppression: same `ignoreDependencyOverrides` config-only rule as `unused-dependency-overrides`. JSON `actions[]` emits `fix-dependency-override` as primary plus the same `add-to-config` suppress. Filter-to-workspaces clears findings. See also: `unused-dependency-overrides` (the consumer-not-declared variant). See issue #336.
