# fallow-extract crate

Key modules:
- `lib.rs` — Public API: `parse_all_files()` (parallel rayon dispatch, cache-aware), returns `ParseResult` with modules + cache hit/miss statistics. UTF-8 BOM is stripped (via `crate::strip_bom`) at every file-read entry point (`parse_single_file_cached`, `parse_single_file`, `parse_from_content`) and again as defense in depth at the top of `parse_source_to_module`, so `content_hash`, `compute_line_offsets`, and the oxc parser all see the same post-BOM byte sequence; line numbers on BOM-bearing files are not shifted by the codepoint. The fix module's `stage_fixed_content` re-prepends the BOM on write so `fallow fix` round-trips Windows-authored files unchanged. `CACHE_VERSION` bumped from 88 to 89 (issue #475) because pre-fix caches keyed hashes over BOM-included bytes.
- `visitor.rs` — Oxc AST visitor extracting imports, exports, re-exports, members, whole-object uses, dynamic import patterns, namespace destructuring (`const { a, b } = ns` → member accesses)
- `sfc.rs` — Vue/Svelte SFC script extraction (HTML comment filtering, `<script src="...">` support, `lang="ts"`/`lang="tsx"` detection, handles `>` in quoted attributes). Scans `generic="..."` (Vue) / `generics="..."` (Svelte) script-tag attributes for type references via an augmented-source probe so type-only imports consumed only inside the constraint stay classified as type-referenced. Orchestrates template usage tracking via `sfc_template/`.
- `sfc_template/` — Template-visible import usage tracking for Vue and Svelte. Framework-specific scanners (`vue.rs`, `svelte.rs`) parse template markup to detect import references (`{formatDate(x)}`, `utils.formatDate()`). Shared scanner (`scanners.rs`) and helpers (`shared.rs`) provide HTML tag/curly-section parsing and expression analysis.
- `template_usage.rs` — `TemplateUsage` struct and `analyze_template_snippet()` for parsing template expressions via Oxc to extract used bindings and member accesses.
- `astro.rs` — Astro frontmatter extraction between `---` delimiters, plus Astro-processed `<script src="...">` references and inline `<script>` ESM imports from the template body (Astro bundles per-component client scripts at build time, so processed script shapes contribute to reachability; attributed inline scripts such as `is:inline` are left alone).
- `mdx.rs` - MDX import/export extraction with multi-line brace tracking and fenced code block skipping
- `css.rs` — CSS/SCSS directive extraction (`@import`, `@use`, `@forward`, Tailwind `@plugin`, `@apply`, `@tailwind`) and CSS Module class name extraction (`.module.css`/`.module.scss` → named exports)
- `css_in_js.rs`: CSS-in-JS tagged-template lifter (`css_in_js_virtual_stylesheet`, CSS program Phase 3b). The tagged-template analogue of `sfc_css::sfc_virtual_stylesheet`: a lexical byte scanner over JS/TS source that lifts styled-components / emotion / linaria template-literal CSS (`` styled.div`...` ``, `` css`...` ``, `` keyframes`...` ``) into a blank-line-padded virtual stylesheet, masking `${...}` interpolations to a CSS-valid placeholder, so `compute_css_analytics` + styling-health analyze CSS-in-JS the same as `.css` files. Health-time-only over file source (no cache, no `CACHE_VERSION` bump); template-literal form only (object form is `css_in_js_object.rs`). The engine's JS/TS CSS-walk arm (`css_report_scan_target` in `crates/engine/src/health/execute.rs`) is dep-gated on `project_uses_css_in_js`.
- `css_in_js_object.rs`: CSS-in-JS OBJECT-notation lifter (`css_in_js_object_sheets`, CSS program Phase 3c). The AST analogue of `css_in_js.rs`: oxc-parses JS/TS and walks call expressions (`oxc_ast_visit::Visit`) for object-notation style calls (vanilla-extract `style`/`globalStyle`/`styleVariants`/`recipe`, emotion `css`/`styled.x`, Panda `css`/`cva`, StyleX `stylex.create`), serializing each style OBJECT into the same blank-line-padded `.fallow-css-in-js{ ... }` virtual stylesheet 3b emits. STATIC-ONLY serialization (camelCase->kebab, implicit-px outside the unitless set with numbers rendered from their parsed value, selector-shaped keys become nested rules); dynamic/spread/computed-key/non-selector-object values are DROPPED, never guessed (no JS interpreter, zero FP). Recognition is gated on IMPORT-BINDING provenance (`build_import_map` maps each binding to its `Lib`; named imports dispatch on the canonical imported name so an alias still fires), so a local `style`/`css` helper or an unrelated `cva` never serializes. Returns THREE sheets (`CssInJsObjectSheets { structural, structural_partial, atomic }`): real-selector non-atomic CSS (full grade + dedup), non-atomic buckets that dropped a dynamic declaration (grade inputs, dedup suppressed by the engine), and flat StyleX/Panda atomic CSS (token-sprawl only, excluded from the structural grade + dedup). Health-time-only, no `CACHE_VERSION` bump. The engine arm (`css_report_scan_items` + `CssGradeScoring` + `StylingScoringInputs` in `execute.rs`) runs it alongside the 3b template lifter and threads the non-atomic declaration split into the grade so atomic CSS never inflates it.
- `html.rs` — HTML asset reference extraction (`<script src>`, `<link rel="stylesheet" href>`, `<link rel="modulepreload" href>` → `SideEffect` imports). Regex-based, comment-stripped.
- `graphql.rs` — GraphQL document `#import "./fragment.graphql"` / `# import '../fragment.gql'` extraction for relative string specifiers. Emits `SideEffect` imports so document-only fragment/schema edges participate in reachability.
- `parse.rs` — File type dispatcher. Runs `oxc_semantic` after parsing to detect unused import bindings (imports where the binding is never read in the file).
- `cache.rs` — Incremental bitcode cache with xxh3 hashing. Unchanged files skip parsing and load from cache. Pruned of stale entries on each run.
- `complexity.rs` — Per-function cyclomatic/cognitive complexity via single-pass `ComplexityVisitor`
- `template_complexity/` — Synthetic `<template>` cyclomatic/cognitive complexity for Angular `.html` templates AND inline `@Component({ template: \`...\` })` decorators, plus Vue and Svelte SFCs. `engine.rs` is the framework-agnostic JS-expression complexity engine (`TemplateComplexity`, `compute_expression_metrics` and its byte-safe tokenization helpers: logical operators, ternary branches, optional chaining) shared by all three scanners, so a `v-if`/`{#if}`/`@if` condition scores identically. `mod.rs` hosts the Angular `TemplateScanner` + `compute_angular_template_complexity` and the shared `build_template_complexity` emission shape (drop-when-trivial, anchor at first non-trivial expression, build `FunctionComplexity { name: "<template>", ... }`). The Angular scanner covers control-flow blocks (`@if`/`@else`/`@for`/`@switch`/`@case`/`@defer (when ...)`/`@let`), legacy structural directives (`*ngIf`, `*ngFor`), bound attributes (`[x]`, `(x)`, `bind-x`, `on-x`), and `{{ }}` interpolations. `vue.rs` / `svelte.rs` host the SFC scanners (`compute_vue_template_complexity` / `compute_svelte_template_complexity`): each masks `<script>`/`<style>`/`<!-- -->` (replaced with equal-length spaces via `mask_ranges`, so SFC `<script>` control flow is NOT double-counted, it is scored by `translate_script_complexity`) and scans the template region. Vue counts `v-if`/`v-else-if`/`v-else`/`v-for`/`v-show` (incl. `<template v-for>`) as control flow; Svelte counts `{#if}`/`{:else if}`/`{#each}`/`{#await}`/`{:then}`/`{:catch}`/`{#key}`. All three also score bound-attribute / interpolation EXPRESSION complexity (logical ops, ternaries, optional chaining): Vue on `:`/`@`/`v-bind`/`v-on`/`v-model` directive values + `{{ }}`, Angular on `[x]`/`(x)` bound attributes + `{{ }}`, and Svelte on `{ }` text interpolations AND tag-interior attribute bindings (`class={cond ? a : b}`, `onclick={x && y}`, `class:active={...}`; Svelte's `scan_element` scans `{ }` sections inside the tag rather than skipping it, quote-aware so a `>` in an attribute value does not end the tag early). Both Vue and Svelte track cognitive nesting from CONTROL-FLOW depth, NOT raw markup depth: only a control-flow-bearing element/block opens a nesting level for its subtree (Vue keeps a per-element `tag_stack` so plain `<div>` wrappers do not shift depth; Svelte skips HTML tags for nesting), so a `v-if`/`{#if}` buried under plain markup is not over-weighted, matching Angular's `block_depth` and the cognitive-complexity standard. A bare `v-else` / `{:else}` is a continuation (cognitive +1, no cyclomatic), matching Angular's bare `@else`. Malformed templates short-circuit via a unit `ScanError` and produce no finding rather than recovered noise. Angular external templates are invoked from `parse_html_to_module_with_complexity` for `.html` files; inline templates from `parse::append_inline_template_complexity` (the visitor records each captured `template:` literal in `ModuleInfoExtractor::inline_template_findings`, then the scanner remaps the resulting line/col onto the `@Component`/`@Directive` decorator's start so jump-to-source lands on the decorator and `// fallow-ignore-next-line complexity` placed directly above suppresses via the existing health-side check). Vue/Svelte are invoked from `sfc::parse_sfc_to_module` under `need_complexity`, appending the synthetic entry to `combined.complexity` alongside the script functions. Template literals containing `${...}` expressions and `template:` properties bound to a variable are skipped (out of scope for the first cut).
- `tests/` — Integration tests split by parser type: `js_ts.rs`, `sfc.rs`, `astro.rs`, `mdx.rs`, `css.rs`
