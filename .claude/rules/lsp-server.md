# fallow-lsp crate

Key modules:
- `main.rs` — LSP server setup, `LanguageServer` trait impl, event handling
- `diagnostics/`: Diagnostic generation: `mod.rs` (dispatch), `unused.rs`, `structural.rs`, `quality.rs`, `security.rs`. `security.rs` (issue #891) surfaces `AnalysisResults.security_findings` as opt-in `INFORMATION`-severity diagnostics (the LSP translation of the CLI's `[I]` advisory glyph for unverified candidates; fixed, NOT mapped from rule severity, matching every other block's fixed-per-kind severity). Opt-in is automatic: both security rules default `off` and the LSP reuses the project config, so `security_findings` is empty (zero diagnostics) unless the user raises `rules.security-sink` / `security-client-server-leak` to `warn`/`error`. Per-kind `code` token (`security-sink` / `security-client-server-leak`, matching the suppression tokens); absolute-path URI built directly (paths are absolute internally, no `root.join`). `security_diagnostic(finding)` is shared by `push_security_diagnostics` and the suppress code action so the published diagnostic and the action's linked diagnostic correlate exactly. The two `DIAGNOSTIC_ISSUE_TYPES` entries (`config_key` == the rule names) wire the publish-time disable filter and the `fallow/issueTypes` request.
- `code_actions/` — Quick-fix and refactor code actions
- `code_lens.rs`: reference-count Code Lens above export declarations, plus a descriptive React component-intelligence summary lens above each component (`rendered Nx (M parents), P props, H hooks (breakdown)`), gated on the editor `collect_usages` path so it is LSP-only and never runs on the CLI / `audit` path.
- `hover.rs`: hover showing export usage, unused status, duplicate block locations, and React per-prop usage (`read in body` / `passed from N call sites`, with an optional `forwarded N levels: A > B > C` prop-drilling trace) plus a component summary hover. The React intel rides the in-process serde-skip `AnalysisResults.react_component_intel` carrier (gated on `collect_usages`); per-prop position matching uses the same half-open `[col, col+len)` convention as the finding hovers, and a real finding hover wins over the descriptive intel.
- `markdown.rs` — `format_inline_code(value)` helper for embedding user-controlled identifiers (export names, member names, import specifiers, file paths) into hover `MarkupKind::Markdown` bodies. Wraps the value in a CommonMark inline code span with a backtick-fence length one greater than the longest backtick run in the value (with space padding when the value starts or ends with a backtick). Inside the code span CommonMark suppresses every inline construct (links, emphasis, images, command URIs), so the value renders verbatim and cannot leak markdown syntax. Every interpolation into hover MarkupContent must go through this helper. `Diagnostic.message` is plain text per the LSP spec and must NOT be markdown-escaped: VS Code's "Fix all in file" correlation matches the published diagnostic by `range + message + code` equality, and escaping breaks the match.
- `position.rs`: `PositionMapper`, the byte-to-UTF-16 conversion at the LSP protocol boundary. Analysis results (`EditorAnalysisResults`) carry 0-based BYTE columns (`byte_offset_to_line_col` in `crates/types/src/extract.rs`); the negotiated LSP wire encoding is UTF-16 (`server_capabilities.rs` advertises `positionEncoding: utf-16`). EVERY emitted `Position.character` derived from an analysis-result column, and every comparison of a client-supplied `position.character` against one, MUST route through `PositionMapper::utf16_col` / `utf16_col_span`; never pass a byte column or a `.len()` byte width through raw. Exempt by construction: whole-line/whole-file ranges (`character: 0` starts, `character: u32::MAX` ends, `FIRST_LINE_RANGE`) and widths computed from live document text via `encode_utf16().count()` (the `quick_fix.rs` pattern; see the `indent_utf16` fix, main `618f1ddd7b3`, where a raw byte indent corrupted a destructive edit under nbsp indentation). The mapper lazily reads file content from DISK and caches per path, matching the disk state the analysis ran against; unreadable files degrade to byte-column passthrough, never a panic. Complexity/code-lens data produced by `crates/extract/src/inventory.rs::line_col_utf16` is ALREADY UTF-16; do not double-convert it; check a value's provenance before mapping.

## Code-action staleness re-validation

`build_remove_export_actions` re-validates the live document line at `export_line` against the cached `AnalysisResults` before producing a destructive `TextEdit`. A `did_change` between `did_save` (which triggered analysis) and `code_action` could leave the cached line referring to text the user has reshaped in memory. The check uses `contains_identifier_bounded` (helper in the same file) to require non-identifier characters on both sides of the cached export name in the post-prefix substring, defeating substring collisions like cached `foo` matching live `foobar`. `export default` declarations are special-cased: their synthetic export name `"default"` is part of the prefix string and never appears in the post-prefix source, so the prefix check (which already runs) is the validation. `build_remove_catalog_entry_actions` and `build_remove_empty_catalog_group_actions` have equivalent validation via `line_matches_catalog_key`. `build_delete_file_actions` is a whole-file delete with no line dependency.

## Diagnostic publish staleness

`run_analysis` captures a `VersionSnapshot` (`FxHashMap<Uri, i32>`) after acquiring `analysis_guard.try_lock()` and before dispatching `spawn_blocking`. The snapshot reflects the document state the analysis is about to run against. After the join, the snapshot is threaded into `publish_collected_diagnostics`, which reads the live `documents` map ONCE into a local `live_documents` map and consults it per URI via `uri_is_stale`. A URI is treated as STALE under three conditions: (a) present in the snapshot AND live version strictly greater than snapshot (user edited mid-run), (b) present in the snapshot AND absent from live `documents` (user closed via `did_close` mid-run), or (c) absent from the snapshot AND present in live `documents` but the live buffer differs from the on-disk file (opened or edited between snapshot and publish; the analysis did not see the buffer the client now holds). If that opened-mid-run buffer still matches disk, the analysis saw the same text and the URI can update the cache/push path; this covers first-open startup races without requiring a save. Only URIs absent from BOTH snapshot AND live `documents` are NOT stale: cross-file diagnostics anchored to files the user never `did_open`'d via the LSP (`package.json` for unlisted dependencies, `pnpm-workspace.yaml` for catalog references) can be reported normally with `None` for a push version slot. Stale URIs skip client notification + skip cache update, but are still inserted into `new_uris` so the next-run stale-clearing branch does not erase the last-valid diagnostics from the client; the pull-model `cached_diagnostics` cache therefore stays at its last-valid state until the next analysis catches up. Non-stale URIs from the snapshot publish with `Some(snapshot_version)` in `PublishDiagnosticsParams.version` for push-only clients (was always `None` before issue #450), letting LSP 3.17 clients drop already-superseded notifications client-side. Push-suppression keys on whether the client has ACTUALLY pulled (the `client_pulls` flag, set on the first `textDocument/diagnostic` request), NOT on the advertised `refreshSupport` capability. This matters because refresh-capable clients can still choose not to issue `textDocument/diagnostic` for a given document; keying delivery on the capability silently blanked open-file diagnostics for such clients. Once a client has pulled, open documents update `cached_diagnostics` without sending `textDocument/publishDiagnostics`, and after the cache is updated the server sends `workspace/diagnostic/refresh` so the client re-pulls the current cache without also populating a push-diagnostic namespace. On the FIRST pull, the `diagnostic` handler clears any push-model diagnostics emitted for open documents during startup (the first `run_analysis` fires from `initialized`, before any pull) so they do not double with the pull namespace. Unopened-file diagnostics are still pushed even to pulling clients because Fallow does not implement `workspace/diagnostic` and clients such as Neovim only pull diagnostics for open buffers. When a file is opened by a client that has already pulled, `did_open` sends an empty `publishDiagnostics` notification for that URI to clear any earlier unopened-file push, then sends `workspace/diagnostic/refresh` so the client pulls the cached diagnostics without requiring a save. The stale-clearing branch at the end of `publish_collected_diagnostics` mirrors the same stale skip and removes old cache entries; push-only clients and unopened files also receive an empty `publishDiagnostics` clear. It reads live documents BEFORE acquiring `cached_diagnostics.write()` to avoid holding a write-lock across an `await` on `documents.read()`. The `fallow/analysisComplete` notification fires once per `run_analysis` invocation regardless of per-URI skips: AI agents reading the wire still see the "analysis done" signal even when some URIs were stale-skipped. The pull-model `RelatedFullDocumentDiagnosticReport` returned by the `diagnostic` handler does NOT carry a `result_id` today; clients that pull during a stale window get the last-valid cached entry. Code-action staleness re-validation (above) continues to defend the destructive-edit path; tightening it with a version compare alongside the cached `AnalysisResults` is a separate hardening task.

## initializationOptions
- `issueTypes` (object): per-issue-type toggles using kebab-case keys, mapped to diagnostic codes by `DIAGNOSTIC_ISSUE_TYPES`. Disabled types are filtered out before publishing. The same catalog is exposed to editor clients through the custom `fallow/issueTypes` request so UI labels and fallback lists can stay in sync with LSP-emitted diagnostics. A second custom request, `fallow/refreshDiagnostics`, lets the editor ask the server to re-drive `workspace/diagnostic/refresh` on demand (calls `spawn_diagnostic_refresh()`, gated on `client_pulls`); the VS Code mute toggle uses it so un-hiding re-renders open-file pull diagnostics via the server's `getAllProviders()` fire rather than the client's per-document `getProvider` match, which can silently no-op (discussion #287). No-op for push-only clients.
- `changedSince` (string): git ref. When non-empty, results and duplication reports are filtered to files changed since the ref via `fallow_core::changed_files::{try_get_changed_files_with_toplevel, filter_results_by_changed_files, filter_duplication_by_changed_files}`. Mirrors the CLI's `--changed-since`. Resolution runs inside `spawn_blocking`; success/failure is surfaced via `log_message`. Path joins use the canonical git toplevel resolved via `git rev-parse --show-toplevel` (cached on the server), so the filter works correctly when the LSP workspace is a subdirectory of the git repo (issue #190).
- `duplication` (object): VS Code forwards the current duplication settings as camelCase fields (`mode`, `threshold`, `minTokens`, `minLines`, `minOccurrences`, `skipLocal`, `crossLanguage`, `ignoreImports`). The LSP merges those values over project config before calling `find_duplicates_in_project`, matching CLI override semantics for scalar fields and preserving config-only fields such as ignore globs.
- `production` (boolean, optional): production-mode override (issue #1055). `true`/`false` force production on/off; a missing or non-boolean key defers to the project config. Read by `initialization_production_override` and applied in `analyze_project_root` by overriding `ResolvedConfig.production` after `config_for_project`, so it affects dead-code, duplication, and inline-complexity alike. Mirrors the CLI-driven sidebar's `--production` / `--no-production`, so the editor squiggles and the sidebar tree agree. VS Code's `fallow.production` enum maps `"on"`/`"off"` to `true`/`false` and omits the key for `"auto"`; the LSP contract is a plain boolean, so a non-VS-Code client sending the string `"on"` gets the safe defer fallback. Read only at `initialize` (VS Code lists `fallow.production` in `RESTART_CONFIG_KEYS`, so a change restarts the server); there is no `didChangeConfiguration` handler.

## Diagnostic.data
When the `changedSince` filter is active, every published `Diagnostic` carries `data: { "changedSince": "<git_ref>" }` (standard LSP `Diagnostic.data` slot). Set centrally via `attach_changed_since_data` after `build_diagnostics`. AI agents reading via `vscode.languages.getDiagnostics()` can use this payload to verify the filter is on and avoid acting on baseline-excluded findings. No `data` is set when the filter is unset.

Security-candidate diagnostics carry `data: { "security": { "kind", "category", "cwe", "sourceBacked", "reachableFromEntry", "blastRadius", "crossesBoundary" } }` (camelCase, `reachableFromEntry`/`blastRadius`/`crossesBoundary` are `null` when `reachability` is absent), set per-diagnostic in `security_data` (`diagnostics/security.rs`). The whole point is agent-legibility off the wire: an agent reading `getDiagnostics()` triages by source-backed-ness and entry-reachability without a hover round-trip or a CLI re-run (the security feature is "feed the agent", so the diagnostic must carry the triage facts, not just a code + message). `attach_changed_since_data` merges `changedSince` into this object (the `Some(Object)` arm), never clobbering it.

Circular-dependency diagnostics also carry `data: { "circularDependency": { "cycleId": "cycle:<hex>", "fileCount": N } }`, set per-diagnostic in `push_circular_dep_diagnostics` (`diagnostics/structural.rs`). Each file in a cycle gets its own diagnostic, anchored at the import edge that points to the next file (`CircularDependency.edges[i]` carries the `path`/`line`/`col` of that hop, computed in core via `cycle_edge_line_col`; the LSP cannot recompute it because it has no module graph). The shared `cycleId` (FNV-1a over the sorted file set, rotation-independent) lets clients fold the N per-file squigglies back into one cycle. The per-edge `Url::from_file_path` is a render-only filter (an unopenable path is skipped from squiggling but never dropped from the `edges` data, so `edges.len() == files.len()` always holds). Cycles whose data has empty `edges` (historical baselines) fall back to the prior single-first-file diagnostic. `attach_changed_since_data` merges `changedSince` into this existing object rather than clobbering it.

## Server capabilities
`build_server_capabilities(advertise_pull_diagnostics)` (in `main.rs`) is the single source of truth for the `ServerCapabilities` returned from `initialize`. It always advertises `text_document_sync`, `code_action_provider`, `code_lens_provider`, and `hover_provider`. It advertises `diagnostic_provider` (LSP 3.17 pull-model) only when the client advertises `workspace.diagnostics.refreshSupport = true` via the plural `diagnostics` capability, so pull-capable clients can pull. The actual push-vs-pull DELIVERY decision, however, keys on whether the client has issued a real `textDocument/diagnostic` (the `client_pulls` flag), NOT on the advertised capability: a client that advertises refresh support but never pulls keeps receiving push diagnostics for open files. Clients that do pull use pull diagnostics for open documents and push for unopened files, avoiding duplicate push/pull diagnostic namespaces in Neovim while preserving workspace-wide findings without a `workspace/diagnostic` handler. Clients that never pull stay push-only so empty `publishDiagnostics` clears remove stale diagnostics immediately. For the pull provider, `inter_file_dependencies = true` (cross-file findings like unused exports / unused dependencies) and `workspace_diagnostics = false` (no `workspace/diagnostic` handler).

## Single-root analysis
`find_project_roots` returns exactly one root: the canonicalized workspace root. The LSP analyzes that root ONCE over the whole tree, matching the CLI (`fallow check` loads one config via `find_and_load(root)` and runs one `analyze_full` pass). `analyze_full` is already workspace-aware: it discovers every workspace package and runs `run_workspace_fast` per package for plugin and script detection, so the single pass covers all sub-package source files, all per-package plugin configs, and full cross-package reachability. `merge_results` / `merge_duplication` fold that single result into the accumulator; `merge_results` is a thin wrapper over `AnalysisResults::merge_into`, the field-exhaustive union the `merge_results_covers_all_fields` test pins against `AnalysisResults` field drift (issue #444).

Before issue #971 the LSP returned the workspace root plus every sub-package and re-ran the entire pipeline per root, then collapsed the cross-root duplicates with a `dedup_results` pass. That re-walked overlapping files once per root (CPU + allocation amplifier on monorepos), and analyzing a sub-package in isolation lost cross-package reachability, so it surfaced false-positive `unused-export` findings (an export used only by a sibling package) that the root pass resolves; the dedup kept-first on the root-sorted-first list already discarded the per-package config results for overlapping files, so per-package `.fallowrc.json` was mostly inert. Single-root removes the redundant work and the isolation false positives, and keeps the LSP in agreement with the CLI. A sub-package's own `.fallowrc.json` still applies when that sub-package folder is opened directly as the workspace (its folder becomes the single root).
