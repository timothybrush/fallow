# VS Code internals reference

Detailed extension architecture, diagnostics, binary lifecycle, generated
contracts, settings, and historical implementation guidance.

Wraps the `fallow-lsp` binary with additional UI features. TypeScript codebase bundled with rolldown.

## Architecture
- `src/extension.ts` , Activation, command registration, lifecycle
- `src/client.ts`: LSP client setup (stdio transport, language selector for JS/TS/Vue/Svelte/Astro/MDX/JSON), wires `DiagnosticFilter` middleware
- `src/diagnosticFilter.ts`: client-side mute filter and severity renderer; caches last unfiltered diagnostics per URI, drops fallow-source diagnostics whose `code` is muted, then renders surviving fallow diagnostics at `fallow.diagnostics.severity`; serves both push (`handleDiagnostics`) and pull (`provideDiagnostics`) middleware paths. On a mute/severity change `refresh()` re-publishes the cached diagnostics into the PUSH `DiagnosticCollection` (covering unopened-file findings such as `package.json` / `pnpm-workspace.yaml`) AND invokes `FilterClient.refreshPullDiagnostics` to force VS Code to re-pull open documents. The pull re-pull is mandatory because VS Code consumes open-file diagnostics via the LSP 3.17 pull path (since the push-disable override was removed), whose collection is owned by vscode-languageclient and untouched by `client.diagnostics.set`; without the re-pull, undoing a hide left open files stuck hidden until the next edit. `client.ts::triggerPullDiagnosticRefresh` implements the re-pull by firing each open-document pull provider's `onDidChangeDiagnosticsEmitter` (`getFeature(DocumentDiagnosticRequest.method).getProvider(doc)`), the same mechanism the server's `workspace/diagnostic/refresh` drives; one fire re-pulls every matching open doc, and it is a no-op for push-only clients or when no fallow document is open. The `refreshPullDiagnostics` adapter ALSO calls `client.ts::requestServerDiagnosticRefresh`, a fire-and-forget `fallow/refreshDiagnostics` custom request whose server handler re-drives `workspace/diagnostic/refresh` via `getAllProviders()` (UNCONDITIONAL, not the per-document `getProvider` selector match), the path proven to re-render after analysis and on `did_open`. This closes discussion #287: the client-side `getProvider(doc)` fire can silently match nothing (selector skew / timing), leaving un-hide stuck until the next edit; the server round-trip is the reliable path and the local fire is the fast-path / older-server fallback (the request rejects `MethodNotFound` on pre-`fallow/refreshDiagnostics` binaries, which is swallowed). The guaranteed escape hatch is the `fallow.resetDiagnosticFilters` command ("Fallow: Reset Hidden Findings (Restart Server)"): it clears all mute state and restarts the client (fresh `did_open` for every open file). Mute state persists in `workspaceState`, so it survives uninstall/reinstall and deleting `.fallow`; `extension.ts` shows a one-time-per-activation prompt with a "Show all findings" action when `isMutedAll()` is true at startup so a stuck-hidden workspace is discoverable. `attachClient` receives an adapter (`{ get diagnostics, refreshPullDiagnostics }`), not the raw client
- `src/diagnosticMute.ts`: `LanguageStatusItem` (right-gutter, severity Warning when anything is muted), QuickPick manage UI (`createQuickPick` + `canSelectMany`), CodeAction provider (`source.fallow.mute` quick-fix), per-category and global toggle commands
- `src/download.ts` , Binary auto-download from GitHub releases (5 platform targets). Installs are serialized across windows by a cross-process lock (`withInstallLock`, atomic `openSync(...,'wx')` mutex in the shared `bin/` dir, with a stale-mtime steal and lock-free degrade) plus a `getManagedBinaryPath` double-check inside the lock, so simultaneous post-release restores install once instead of racing. Each binary is published atomically: download + verify on a unique temp path, publish the `.sig`/`.sha256` sidecar to its final path, then `renameIntoPlace` the binary LAST (so a reader never sees a binary without its sidecar). `renameIntoPlace` treats a locked-but-byte-identical target as success (Windows EPERM/EBUSY from issue #1091).
- `src/commands.ts` , Analysis and fix commands (spawns `fallow` CLI via execFile)
- Tree view providers for dead code (by issue type) and duplicates (by clone family)
- `src/healthTreeView.ts`, `src/health-utils.ts`, `src/health-labels.ts`: the **Health** view (`fallow.health`). Renders Score+grade, Complexity (top N, click opens file:line), Hotspot candidates, and Refactoring candidates from a **separate, lazy** `fallow health --format json` spawn (`runHealthAnalysis` in `commands.ts`). Pure helpers (`buildHealthArgs`, `formatScoreLabel`, `gradeIcon`/`gradeThemeColor`, `severityIcon`, `topPenalties`, `countHealthItems`) live in `health-utils.ts` and are unit-tested; section labels/icons in `health-labels.ts` (kept out of `labels.ts` since these are not `fallow.issueTypes.*` keys). Hotspots and refactoring targets are framed as heuristic candidates pending verification; score and complexity are framed plainly.
- `src/generated/output-contract.d.ts`: **autogenerated** TypeScript shapes for fallow's JSON output. Regenerated from `docs/output-schema.json` by `scripts/codegen-contracts.mjs`. NEVER edit by hand. The current chain is `Rust struct (crates/types, crates/core/duplicates, crates/types/output) -> docs/output-schema.json -> editors/vscode/src/generated/output-contract.d.ts`. For the in-scope finding + duplication + output-augmentation types, the Rust hop is gated by a drift test in `crates/cli/src/bin/schema_emit.rs`: edit the Rust struct, run `cargo test -p fallow-cli --features schema-emit --bin fallow-schema-emit`, reconcile `docs/output-schema.json` against the derived shape (see `CONTRIBUTING.md` "Editing the JSON output contract"), then run `pnpm run codegen:contracts`. CI runs `pnpm run check:contracts` to fail the build when the committed generated files disagree with regen output.
- `src/settings.ts`, `src/labels.ts`, `src/fix-types.ts` , small hand-written modules for content NOT in the JSON output contract: VS Code setting shapes (`IssueTypeConfig`, `DuplicationMode`, `TraceLevel`), UI tree-view labels (`IssueCategory`, `ISSUE_CATEGORY_LABELS`), and the still-unschematized `fallow fix --format json` shape (`FixAction`, `FallowFixResult`).
- `src/types.ts` , thin re-export barrel. Schema-derived types flow from `./generated/output-contract.js`; hand-written types flow from the three modules above. Legacy `FallowCheckResult` / `FallowDupesResult` / `FallowCombinedResult` aliases are preserved for existing consumers; new code should prefer the schema names (`CheckOutput`, `DupesOutput`, `CombinedOutput`).

## Binary resolution order
1. `fallow.lspPath` setting (explicit path, always wins)
2. Local `node_modules/.bin/` in workspace root (devDependency install)
3. `fallow-lsp` in system `PATH`
4. Previously downloaded binary in extension global storage
5. Auto-download from GitHub releases (if `fallow.autoDownload` enabled)

## Key behaviors
- **Lazy CLI analysis** , deferred until sidebar is first made visible (avoids double analysis with LSP)
- **Lazy, opt-out health spawn**: the Health view runs a SEPARATE `fallow health` spawn (never folded into the combined run, which keeps `--skip health`). It fires only when the Health view is first revealed, via its own visibility latch (`healthAnalysisRan`) and `onDidChangeVisibility`, independent of the dead-code/dupes trigger. So opening the editor or the existing two views triggers zero health work (#902 latency isolation). `fallow.health.enabled = false` disables the spawn entirely; `fallow.health.hotspots = false` drops the git-churn walk; `fallow.health.topFindings` caps serialized complexity findings. Health settings live in `REANALYSIS_CONFIG_KEYS`-style handling but route to a health-only re-run, never an LSP restart (`fallow.health.*` are NOT in `RESTART_CONFIG_KEYS`). The status bar gains an optional text-only `health B (82)` segment (gated by `fallow.health.statusBar`).
- **LSP notification** , custom `fallow/analysisComplete` for real-time status bar updates
- **Config watch** - restarts LSP when `fallow.lspPath`, `fallow.configPath`, `fallow.issueTypes`, `fallow.changedSince`, `fallow.duplication`, `fallow.production`, `fallow.autoDownload`, or `fallow.trace.server` changes (`fallow.production` is in `RESTART_CONFIG_KEYS` because it is forwarded via `initializationOptions.production`, which the LSP only reads at startup)
- **Large buffer** , 50MB maxBuffer for CLI output on large monorepos
- **Diagnostic mute and severity**: client-side filter backed by the LSP issue-type catalog. On startup the extension requests `fallow/issueTypes`; if the request fails (older LSP or invalid response), it falls back to the bundled `DIAGNOSTIC_CATEGORIES` list in `diagnosticFilter.ts`. Muting remains instant and does not restart the LSP. State lives in `context.workspaceState` under `fallow.diagnosticFilter.v1`. Severity rendering is a global editor setting, `fallow.diagnostics.severity` (`warning`, `information`, `hint`), applied after mute filtering and refreshed live from cached diagnostics without restarting the LSP or re-running CLI analysis. Filter ALWAYS guards on `Diagnostic.source === "fallow"` so TypeScript / ESLint diagnostics flow through untouched. Cache is bounded by `onDidCloseTextDocument` eviction. Keep the fallback list in sync with `DIAGNOSTIC_ISSUE_TYPES` / `fallow/issueTypes` in `crates/lsp/src/main.rs` plus diagnostics emitted outside the issue-type catalog; vitest coverage flags drift.
- **Diagnostics on/off status-bar toggle** (`diagnosticStatusBar.ts`): an always-visible left status-bar item (priority 48; main 50, audit 49) shown by default and gated by `fallow.diagnostics.statusBar`. It reads `$(eye) Fallow` when findings are visible and `$(eye-closed) Fallow: hidden` when all are muted (no background color, mirroring the main item, NOT the audit item's warning tint), and reuses the existing `fallow.toggleAllDiagnostics` command. It subscribes to `DiagnosticFilter.onDidChange`, so it stays in sync with the right-gutter Language Status item, the Manage quick pick, and code-action mutes; `createDiagnosticStatusBar` is idempotent (disposes the prior item + subscription first). Created/disposed LIVE on the setting change via `syncDiagnosticStatusBar` in `extension.ts` (mirrors `syncAuditStatusBar`), with a teardown disposer in `deactivate`. User-facing wording for the visibility actions is "hide / show" across the button, command titles, the Manage quick pick, the code action, and status messages (the right-gutter `summaryText`/`muteScopeTooltip` already used "hide/hidden"); internal API keeps `muted*` / `toggleMutedAll` / `STATE_KEY` and the command IDs (`fallow.toggleAllDiagnostics`, `fallow.manageDiagnosticMutes`, etc.) unchanged.

- **Complexity lens + progressive-disclosure breakdown** (`complexityLens.ts`, `complexityDecorations.ts`): `ComplexityLensProvider` renders one code lens per complex function at its signature line, read from the Health view's cached findings, titled `<name>: N cyc, N cog · show/hide breakdown`. The title flips with the function's UNIFIED expansion state (`controller.isExpanded` = pinned OR Health-view-selected), so a selected function's lens reads `hide breakdown` and the sidebar selection and the lens never disagree; clicking fires `fallow.toggleComplexityBreakdown` -> `ComplexityDecorationController.toggleExpanded`, which collapses fully (drops the pin AND clears the selection contribution) when expanded, else pins. Gated on `fallow.health.inlineComplexity` (default true). The dense per-line `+N` DETAIL decorations render ONLY for expanded functions: pinned (lens click) UNION selected (Health-view `onDidChangeSelection` -> `setSelectedFunction`, cleared on deselect / non-complexity selection / view hidden), or every function when `fallow.complexity.afterText` (default false) is on. Hover is a `registerHoverProvider` delegating to `controller.provideHover` (function summary on the signature line, per-line breakdown on a decision-point line), DECOUPLED from the decoration so peek works with `afterText` off. The controller's `onDidChange` event is the lens provider's `onDidChangeCodeLenses`, fired on findings / pin / selection / staleness so titles flip and lenses clear in step; an edited document is marked stale and suppresses decorations + lenses + hovers until the next health run. **The extension OWNS the lens (so it can toggle); `client.ts` no longer forwards `health.inlineComplexity` to the LSP. The editor-agnostic LSP complexity lens (server default OFF, opt-in via `initializationOptions.health.inlineComplexity`) stays available to other editors (Neovim/Zed/Helix) with no double-render in VS Code.** The toggle command is invokable and bindable; there is no hover trigger on the lens itself (VS Code has no per-lens hover event), so peek lives on the code line.

## Settings
`fallow.lspPath`, `fallow.autoDownload`, `fallow.issueTypes`, `fallow.duplication.threshold`, `fallow.duplication.minOccurrences`, `fallow.duplication.mode`, `fallow.production`, `fallow.changedSince`, `fallow.health.enabled`, `fallow.health.hotspots`, `fallow.health.topFindings`, `fallow.health.statusBar`, `fallow.health.inlineComplexity`, `fallow.complexity.breakdownEnabled`, `fallow.complexity.afterText`, `fallow.complexity.decorationCap`, `fallow.diagnostics.statusBar`, `fallow.diagnostics.severity`, `fallow.trace.server`

`fallow.health.inlineComplexity` (default true) gates the extension-rendered complexity lens; `fallow.complexity.breakdownEnabled` (default true) is the master switch for the whole breakdown feature (lens + detail + hover); `fallow.complexity.afterText` (default false) forces the per-line `+N` detail on for every complex function instead of only expanded ones; `fallow.complexity.decorationCap` (default 200) caps findings fetched for decoration. None restart the LSP; changing `inlineComplexity` / `afterText` re-renders live via the controller's `onDidChange`.

`fallow.health.*` gate the Health view's separate spawn (see "Lazy, opt-out health spawn" above): `enabled` (default true) toggles the view and the spawn; `hotspots` (default true) toggles the git-churn section; `topFindings` (default 20) caps complexity findings via `--top`; `statusBar` (default true) shows the score/grade segment. Changing any of them re-runs only the standalone health spawn (or, for `statusBar`-only, re-renders from the cached report). They never restart the LSP or re-run the combined sidebar analysis.

`fallow.changedSince` mirrors the CLI's `--changed-since`: when set to a git ref, the LSP scopes diagnostics (and the CLI-driven sidebar) to files changed since that ref. Forwarded via `initializationOptions.changedSince` and as `--changed-since <ref>` to the CLI in `commands.ts`. Empty string means full scope. Changing the setting restarts the LSP and re-runs the sidebar analysis.

`fallow.duplication.*` settings are forwarded to the LSP through `initializationOptions.duplication` and to CLI analysis through `commands.ts`. Changing any duplication setting restarts the LSP and re-runs the sidebar analysis. `fallow.production` and all `fallow.duplication.*` settings declare `scope: "resource"` so a stale global value cannot silently override a project's committed `.fallowrc.json` / `.vscode/settings.json` (issue #1055).

`fallow.production` is a tri-state enum (`"auto"` | `"on"` | `"off"`, default `"auto"`) read via the inspect-based `getProductionOverride()` (returns `boolean | undefined`; a legacy stored boolean still maps to on/off). `"auto"` defers to the project config on every surface; `"on"` forwards `--production` to the sidebar and `production: true` to the LSP; `"off"` forwards `--no-production` to the sidebar (version-gated at CLI 2.90.0, degrades to defer on older binaries) and `production: false` to the LSP. The two editor-diagnostics surfaces (sidebar tree + LSP squiggles) therefore always agree. Force-off is intentionally scoped to those two surfaces: `buildAuditArgs` / `buildHealthArgs` / `buildCoverageArgs` / `buildFixArgs` still forward `--production` only on `"on"` and treat `"off"` as defer (they widened their `production` field to `boolean | undefined` but kept the truthy check).

## Development
```bash
cd editors/vscode
pnpm install
pnpm run codegen:contracts   # regenerate VS Code and npm generated contracts
pnpm run check:contracts     # CI-equivalent drift check; fails when committed files disagree with regen
pnpm run build     # rolldown production bundle (runs codegen via prebuild)
pnpm run watch     # development watch mode
pnpm run lint      # tsc --noEmit
pnpm run test      # unit + integration tests (vitest)
pnpm run package   # vsce package --no-dependencies (runs codegen via prepackage)
```

`prebuild` and `prepackage` both invoke `codegen:contracts`, so `pnpm build` and `pnpm package` always bundle against fresh contracts. `pnpm watch` does NOT run codegen; if you change the schema during a watch session, re-run `pnpm run codegen:contracts` manually.

## Version management
Extension version is set from the git tag by CI , do not manually update `editors/vscode/package.json` version. The release workflow handles everything.

## Engine floor
`engines.vscode` is pinned at `^1.96.0` and `@types/vscode` at `1.96.0`. Dependabot is told to ignore `@types/vscode` in `.github/dependabot.yml`. The combination keeps the VSIX installable on Cursor and Windsurf (older VS Code forks) and satisfies the `vsce` constraint that `@types/vscode` must be `<=` the engine floor. `test/engine-floor.test.ts` asserts both. Do NOT raise either version without a deliberate reason. A `@types/vscode` bump alone breaks `vsce package`; raising `engines.vscode` breaks install on every fork below that base. The cautionary tale is commit `32685cda`, which bumped to ^1.116.0 just to align with a Dependabot types PR.
